# 외부 연동 TLS 전환에서 인터페이스와 설정을 같이 맞춘 기록

## 이번에 한 작업

외부 시스템 연동 구간에 TLS 통신 옵션이 추가되면서, 기존 연동 래퍼와 네이티브 호출부를 같이 수정했다.

처음에는 라이브러리 파일만 교체하면 될 것처럼 보였지만, 실제로는 초기화 인터페이스, 설정값, 호출 인자, 배포 파일까지 같이 맞춰야 했다.

내가 실제로 손댄 범위는 다음과 같았다.

- 외부 연동 래퍼의 초기화 함수 인자 변경
- 네이티브 함수 호출부의 매개변수 순서와 개수 정리
- TLS 사용 여부를 설정 파일에서 켜고 끌 수 있게 추가
- HTTPS 통신에 필요한 런타임 라이브러리 교체
- 기존 비 TLS 환경에서도 동작하도록 기본값 확인
- 배포된 라이브러리 버전과 애플리케이션 호출부가 맞는지 검증

## 구현하면서 조심한 점

가장 위험한 부분은 "라이브러리는 최신인데 애플리케이션은 예전 인자 규칙으로 호출하는 상태"였다.

이 경우 컴파일은 통과해도 런타임에서 초기화가 실패하거나, 더 나쁘게는 잘못된 인자가 전달될 수 있다. 그래서 단순히 파일 교체로 보지 않고 호출 계약을 다시 맞추는 작업으로 봤다.

## 설정 플래그를 둔 이유

TLS 기능은 한 번에 모든 환경에 강제로 켜기 어려웠다.

운영 환경마다 외부 시스템의 설정, 인증서 준비 상태, 네트워크 정책이 다를 수 있기 때문이다. 그래서 설정 파일에 TLS 사용 여부를 두고, 기능을 켤 수 있는 환경과 기존 방식으로 유지해야 하는 환경을 나눴다.

```text
tls.enabled=true
tls.enabled=false
```

실제 설정명은 공개하지 않지만, 방향은 이랬다. 코드에서는 이 값을 읽어 초기화 단계에서 외부 연동 래퍼에 전달했다.

## 검증한 흐름

검증은 세 단계로 나눴다.

1. TLS 옵션이 꺼진 상태에서 기존 연동이 유지되는지 확인
2. TLS 옵션이 켜진 상태에서 초기화와 로그인 흐름이 통과하는지 확인
3. 라이브러리 파일이 잘못 배포됐을 때 오류가 명확히 보이는지 확인

특히 설정값만 바꾸고 재기동했을 때 어떤 코드 경로를 타는지 로그를 확인했다. TLS 전환 작업은 오류가 나면 네트워크 문제처럼 보이기 쉬워서, 초기화 단계와 통신 단계의 실패를 분리해서 봐야 했다.

## 정리

외부 연동 TLS 전환은 "보안 옵션 하나 추가"가 아니라 애플리케이션과 네이티브 라이브러리 사이의 호출 계약을 다시 맞추는 작업이었다.

앞으로 비슷한 작업을 할 때는 라이브러리 교체, 인터페이스 변경, 설정 플래그, 배포 파일, 기존 환경 호환성을 한 묶음으로 확인해야 한다.
