# 알 수 없는 에러 코드가 로그에 반복될 때 확인할 것

## 상황

기능은 실패했고 로그에는 숫자 형태의 결과 코드가 남았는데, 애플리케이션 쪽에서는 그 코드를 해석하지 못해 "알 수 없는 오류"로만 처리되는 경우가 있다.

예를 들면 이런 형태다.

```text
change result: 1010
Cannot find this err[1010] in enumerate
```

이때 바로 기능 로직만 의심하면 원인을 늦게 찾을 수 있다. 실제 문제는 기능 실패 자체가 아니라, 실패 코드를 만든 계층과 그 코드를 메시지로 바꾸는 계층의 버전이 어긋난 것일 수 있다.

## 먼저 분리해서 볼 것

숫자 코드는 보통 여러 계층을 거쳐 올라온다.

1. 외부 라이브러리 또는 네이티브 코드가 결과 코드를 반환한다.
2. Java 같은 중간 계층이 그 코드를 받는다.
3. enum 또는 메시지 맵에서 사용자에게 보여줄 문구를 찾는다.
4. 화면 또는 로그에 최종 메시지를 남긴다.

따라서 `1010` 같은 코드를 찾지 못했다는 로그는 "1010이라는 실패가 왜 났는가"와 별개로 "1010을 애플리케이션이 아직 모른다"는 뜻이다.

## 체크 순서

가장 먼저 결과 코드가 어디에서 만들어졌는지 확인한다. 네이티브 라이브러리, 외부 SDK, 서버 API, DB 프로시저 중 어느 쪽이 원본인지 알아야 한다.

다음으로 애플리케이션의 코드 매핑을 확인한다.

```java
String message = errorMessages.get(resultCode);
if (message == null) {
    log.warn("Unknown result code: {}", resultCode);
    return ChangeResult.failed("처리되지 않은 오류 코드입니다.");
}
```

이때 중요한 점은 모르는 코드를 성공으로 넘기면 안 된다는 것이다. 코드 의미를 모르면 기본값은 실패로 두고, 상태 변경이 있었다면 롤백 또는 재확인 절차를 타게 해야 한다.

## 흔한 원인

- 외부 DLL 또는 SDK는 업데이트됐지만 Java enum은 업데이트되지 않음
- 문서에는 새 코드가 추가됐지만 애플리케이션 메시지 맵에 누락됨
- 같은 숫자를 모듈마다 다른 의미로 사용함
- 실패 후 검증 로직은 성공했는데 최초 결과 코드만 남아 화면이 실패로 보임

## 대응 방법

새 코드를 바로 메시지 맵에 추가하기 전에 의미를 확인해야 한다. 숫자 위치가 기존 코드 근처에 있다고 해서 의미를 추측해 넣으면 나중에 더 큰 혼란이 생긴다.

확인할 자료는 보통 다음 순서가 좋다.

- 외부 라이브러리 헤더 파일
- SDK 문서
- 네이티브 코드의 `#define` 또는 enum
- 릴리즈 노트
- 실제 실패 재현 로그

의미를 확인했다면 애플리케이션 enum, 화면 메시지, 테스트 데이터를 함께 업데이트한다.

## 예방

결과 코드를 다루는 코드는 "알 수 없는 코드" 자체를 명확한 실패 케이스로 다루는 편이 안전하다.

```java
if (!knownCodes.contains(code)) {
    return Failure.unknownExternalCode(code);
}
```

그리고 새 SDK를 반영할 때는 컴파일 성공만 보지 말고, 결과 코드 목록이 애플리케이션 메시지 맵과 맞는지도 같이 확인해야 한다.

정리하면, "에러 코드가 없다"는 로그는 단순 문자열 누락이 아니라 계층 간 계약이 깨졌다는 신호다. 코드의 의미를 확인하고, 매핑과 실패 처리 정책을 함께 맞춰야 한다.
