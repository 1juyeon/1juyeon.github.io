# 비밀번호 변경 이력 테이블에는 무엇을 남겨야 할까

## 상황

관리 기능에서 비밀번호를 변경하면 이력을 남겨야 할 때가 있다.

처음에는 변경 전 비밀번호, 변경 후 비밀번호, 결과만 저장하면 될 것처럼 보인다. 하지만 운영과 감사 관점에서는 "어떤 방식으로 변경됐는지"도 중요할 수 있다.

예를 들면 다음처럼 구분할 수 있다.

- 사용자가 직접 입력한 값
- 정책에 따라 랜덤 생성된 값
- 복구 절차로 재설정된 값
- 자동 스케줄에 의해 변경된 값

이런 구분을 남기는 컬럼이 `change_type` 같은 역할을 한다.

## change_type이 필요한 경우

`change_type`은 비밀번호 값 자체가 아니라 변경 방식의 의미를 담는다.

| 값 | 의미 예시 |
| --- | --- |
| MANUAL | 사용자가 직접 입력 |
| RANDOM | 정책 기반 자동 생성 |
| RESET | 관리자 재설정 |
| SCHEDULED | 스케줄 변경 |

이 값이 있으면 나중에 "왜 이런 비밀번호가 생겼는지" 추적할 수 있다.

## 필요 없을 수도 있다

반대로 특정 화면에서 변경 방식이 항상 하나라면 `change_type` 컬럼은 사용자에게 의미 없는 정보가 된다.

예를 들어 DB 비밀번호 변경 화면이 항상 규칙 기반 랜덤 생성만 허용한다면, 이력 화면에 매번 "랜덤"이라고 보여주는 것은 정보량이 낮다.

이럴 때는 DB에는 남기더라도 화면에서는 숨기거나, 아예 해당 기능의 이력 테이블에서는 제거하는 편이 더 깔끔하다.

## 이력에 남길 기본 정보

비밀번호 변경 이력은 최소한 아래 정보를 고려한다.

- 변경 시각
- 수행자
- 대상 계정 또는 대상 시스템
- 변경 방식
- 결과
- 실패 사유
- 요청 IP 또는 세션 정보

단, 비밀번호 평문을 이력에 저장하거나 화면에 표시하는 것은 매우 신중해야 한다. 꼭 필요하다면 암호화 저장, 권한 제한, 마스킹 표시, 감사 로그를 같이 설계해야 한다.

## 실패 이력 처리

실패 이력을 어디까지 남길지도 정해야 한다.

- 입력값 검증 실패
- 권한 부족
- DB 변경 실패
- 설정 파일 저장 실패
- 재시작 실패

모든 실패를 다 남기면 이력은 풍부해지지만 민감 정보가 늘 수 있다. 반대로 DB 변경 이후의 실패만 남기면 실제 상태 변경이 있었던 사건 중심으로 관리할 수 있다.

기준은 명확해야 한다. "DB에 영향을 준 시점 이후의 실패만 이력에 남긴다"처럼 정책을 정해두면 화면과 로그가 덜 흔들린다.

## 정리

이력 테이블은 단순 기록용이 아니라 나중에 장애와 감사를 설명하는 자료다.

`change_type` 같은 컬럼은 변경 방식이 여러 개일 때 의미가 있고, 하나뿐이라면 화면에서는 과감히 숨기는 편이 낫다. 대신 결과, 실패 사유, 수행자, 시각처럼 운영자가 실제로 판단하는 데 필요한 정보는 빠뜨리지 않아야 한다.
