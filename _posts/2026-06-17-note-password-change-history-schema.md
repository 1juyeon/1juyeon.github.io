# 민감 설정 변경 이력 화면을 정리하며 결정한 것

## 이번에 한 작업

민감한 운영 설정값을 변경하는 기능을 만들면서 변경 이력 테이블과 화면도 같이 만들었다.

처음에는 이력에 변경 방식까지 보여줄지 고민했다. 예를 들면 직접 입력인지, 랜덤 생성인지, 재설정인지 같은 구분이다.

하지만 이번 화면의 설정 변경은 정책상 규칙 기반 생성 흐름으로 정리되어 있었고, 화면에서 매번 같은 구분값을 보여주는 건 정보량이 낮았다.

그래서 최종 화면에서는 구분 컬럼을 제거했다.

이 화면을 만들면서 실제로 수정한 부분은 다음과 같았다.

- 변경 이력 저장용 테이블과 조회 쿼리 추가
- 성공/실패 결과와 실패 사유를 남기는 저장 흐름 추가
- 최신 5건만 가져오도록 조회 기준 변경
- JSP 테이블에서 구분 컬럼과 페이징 영역 제거
- 비밀번호 보기 버튼은 재인증 후 서버에서 다시 조회하도록 변경
- 컬럼 제거 뒤 남은 `colspan`, JavaScript 함수 참조, count 쿼리 참조를 검색해서 정리

## 최종 이력 화면

화면은 아래 5개 컬럼으로 정리했다.

| 컬럼 | 의미 |
| --- | --- |
| 변경 일시 | 언제 변경했는지 |
| 아이디 | 누가 변경했는지 |
| 이전 비밀번호 | 변경 전 값 |
| 변경 비밀번호 | 변경 후 값 |
| 결과 | 성공/실패와 실패 사유 |

비밀번호 값은 기본적으로 마스킹하고, 필요한 경우에만 보기 버튼으로 확인하도록 했다. 이때도 초기 HTML에 평문을 넣지 않고, 보기 시점에 재인증 후 서버에서 조회하도록 했다.

## change_type을 제거한 이유

DB에는 변경 방식을 남길 수 있지만, 화면에서 보여주는 정보는 사용자가 판단할 수 있어야 한다.

이번 기능에서는 변경 방식이 사실상 하나였기 때문에 `랜덤` 같은 값이 반복해서 표시됐다.

그래서 이력 화면에서는 제거했다.

대신 코드와 DB 구조에서는 나중에 다른 변경 방식이 생길 가능성을 고려해, 필요한 내부 값은 유지할 수 있게 했다.

## 실패 이력 기준

이력에 실패를 어디까지 남길지도 정했다.

입력값 검증이나 규칙 미선택처럼 DB 변경 전에 막힌 오류는 팝업으로만 안내하고 이력에는 남기지 않았다. 아직 실제 상태 변경이 없기 때문이다.

반대로 실제 설정 변경이 시작된 뒤 실패한 경우는 이력에 남긴다.

- DB 계정 변경 실패
- 새 비밀번호 연결 검증 실패
- 설정 파일 기록 실패
- 원복 실패

특히 원복 실패는 수동 조치가 필요하므로 결과 메시지에 명확히 남겨야 한다고 봤다.

## 검증하면서 확인한 것

화면과 컨트롤러를 고친 뒤에는 잔여 참조를 확인했다.

- 페이징 함수 참조가 남아 있지 않은지
- 이력 조회가 최신 5개만 가져오는지
- 구분 컬럼 제거 후 `colspan`이 맞는지
- 실패 행이 결과 메시지로 구분되는지
- 사용하지 않는 count 매퍼가 동작에 영향을 주지 않는지

작은 UI 수정처럼 보여도, JSP에 함수 참조 하나가 남아 있으면 화면 오류가 날 수 있어서 잔여 검색을 꼭 했다.

## 정리

이번 이력 화면 정리는 "많이 보여주는 것"보다 "판단에 필요한 것만 보여주는 것"에 가까웠다.

이력은 감사 자료이기도 하지만 운영자가 장애를 판단하는 화면이기도 하다. 그래서 성공/실패, 실패 사유, 수행자, 시각은 남기고, 의미 없는 반복 컬럼은 덜어냈다.
