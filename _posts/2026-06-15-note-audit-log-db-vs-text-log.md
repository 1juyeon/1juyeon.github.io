## 이번에 한 작업

보안 요구사항을 하나씩 확인하면서 로그인 실패 차단, 장기 미사용자 처리, 중복 로그인, 접근 로그 저장 여부를 코드 기준으로 점검했다.

처음에는 "로그가 남는가"를 서버 로그 파일 기준으로 생각했다. 그런데 요구사항을 따라가다 보니 보안 감사에서 말하는 로그는 단순 텍스트 로그가 아니라 DB에 구조화되어 쌓이고, 관리자가 조회할 수 있는 감사 기록에 더 가깝다는 걸 다시 정리하게 됐다.

그래서 나는 로그를 두 종류로 나눠 봤다.

- 개발자/운영자용 텍스트 로그
- 보안 감사용 DB 이력

이 구분을 하고 나니 구현 여부 판단도 달라졌다.

이때 내가 한 일은 단순히 로그 문자열을 찾는 게 아니었다.

- 로그인 처리 흐름에서 성공, 실패, 로그아웃 이벤트가 어디에 저장되는지 추적
- 실패 횟수와 계정 잠금 정보가 별도 테이블에 남는지 확인
- 장기 미사용자 차단이 스케줄러로 실제 실행되는지 확인
- 세션 중복 로그인 차단이 화면 표시가 아니라 서버 상태로 막히는지 확인
- 관리자 화면에서 감사 기록을 조회할 수 있는지와 권한이 막혀 있는지 다시 구분
- 요구사항 표에는 "저장됨", "화면 없음", "분석 기능 없음"처럼 상태를 나눠 적음

## 코드에서 확인한 것

점검하면서 다음 항목들을 확인했다.

- 로그인 성공/실패/로그아웃/자동 로그아웃 같은 이벤트가 DB 테이블에 남는지
- 로그인 실패 횟수와 잠금 시각이 별도 테이블에 남는지
- 장기 미사용자 차단 스케줄이 실제로 있는지
- 중복 로그인 차단이 세션 캐시 기준으로 동작하는지
- 관리자 화면에서 접근 로그를 날짜별로 조회할 수 있는지
- 일반 사용자가 감사 기록 화면에 접근할 수 없는지

중간에 내가 한 번 잘못 판단한 부분도 있었다. 날짜 검색 UI가 있는 화면을 보고 접근 로그 점검이 가능하다고 생각했는데, 다시 보니 그 화면은 다른 조작 이력을 보여주는 화면이었다.

접속 이력 테이블에는 데이터가 쌓이지만, 그 테이블을 직접 조회하는 화면은 비활성화되어 있었다. 그래서 "DB에는 저장되지만 UI에서 점검할 수 없다"로 정정했다.

## 내가 정리한 기준

이후부터는 보안 로그를 판단할 때 아래 기준으로 나눴다.

| 항목 | 판단 기준 |
| --- | --- |
| 저장됨 | DB 또는 파일에 이벤트가 남는가 |
| 점검 가능 | 관리자가 화면에서 날짜별로 볼 수 있는가 |
| 분석 가능 | 실패 횟수, 월별 집계 등을 자동으로 볼 수 있는가 |
| 권한 통제 | 일반 사용자가 볼 수 없게 막혀 있는가 |
| 감사 추적 | 누가, 언제, 무엇을 했는지 구조적으로 남는가 |

이 기준으로 보면 "DB에 쌓임"은 구현 완료가 아니라 시작점이다. 화면 조회와 권한 제어까지 있어야 점검 가능하다고 말할 수 있다.

## 텍스트 로그와 감사 기록의 차이

텍스트 로그는 장애 분석에는 유용하다.

```text
login failed
user locked
session expired
```

하지만 보안 감사에는 부족하다.

- 보존 정책이 애매할 수 있다.
- 관리자 UI에서 조회하기 어렵다.
- 구조화된 집계가 어렵다.
- 일반 사용자 접근 제한과 연결하기 어렵다.

그래서 보안 요구사항에는 DB 감사 기록이 더 적합하다.

## 정리

이번 작업은 기능을 새로 만든 것보다, 이미 있는 기능을 요구사항 기준으로 다시 판정한 작업에 가까웠다.

가장 큰 정리는 이것이다. "로그가 찍힌다"와 "감사 기록으로 점검 가능하다"는 다르다. 앞으로 보안 요구사항을 볼 때는 저장, 조회, 분석, 권한 통제를 분리해서 확인해야 한다.
