# SSH 기반 서버 계정 변경 기능을 붙이며 나눈 처리 기준

## 이번에 한 작업

서버 계정 변경 기능에서 리눅스 서버 처리를 활성화하고, 여러 배포판과 권한 케이스를 다루도록 보완했다.

기존 흐름은 특정 운영체제와 특정 계정 흐름에 더 맞춰져 있었다. 리눅스 서버까지 다루려면 SSH 접속, 명령 실행, sudo 권한, 추가 사용자 관리, OS 템플릿을 같이 봐야 했다.

내가 실제로 수정한 범위는 다음과 같았다.

- 리눅스 서버 비밀번호 변경 기능 활성화
- 지원 OS 템플릿 추가
- sudo 권한 계정으로 비밀번호 변경 가능하도록 처리
- 추가 사용자 삭제가 누락되는 문제 수정
- SSH 접속 오류 케이스 보완
- 화면 문구와 라이선스 안내 문구 정리

## 구현 흐름

서버 비밀번호 변경은 단순히 `passwd` 명령을 실행하는 문제가 아니었다.

나는 흐름을 이렇게 나눠서 봤다.

1. 대상 서버 OS 유형을 확인한다.
2. 접속 계정과 변경 대상 계정을 구분한다.
3. SSH 접속을 확인한다.
4. sudo 필요 여부를 판단한다.
5. 비밀번호 변경 명령을 실행한다.
6. 결과 메시지와 종료 코드를 해석한다.
7. 추가 사용자 정보와 내부 상태를 동기화한다.

접속 계정과 변경 대상 계정이 항상 같지 않다는 점이 중요했다. 운영 환경에서는 sudo 가능한 계정으로 접속한 뒤 다른 계정의 비밀번호를 바꿔야 하는 경우가 있다.

## 오류 처리를 나눈 이유

SSH 기반 기능은 실패 원인이 다양하다.

- 접속 불가
- 인증 실패
- sudo 권한 없음
- 명령 실행 실패
- OS별 명령 결과 형식 차이
- 변경은 됐지만 결과 파싱 실패

이걸 모두 "비밀번호 변경 실패"로 묶으면 운영자가 조치하기 어렵다. 그래서 가능한 한 접속 단계, 권한 단계, 명령 실행 단계, 결과 확인 단계를 나눠서 보려고 했다.

## 템플릿 관리

리눅스 배포판별로 명령이나 결과 메시지가 조금씩 다를 수 있어서 OS 템플릿도 같이 정리했다.

템플릿을 추가할 때는 단순히 이름만 넣는 것이 아니라, 실제 변경 명령과 결과 판정 기준이 맞는지 확인해야 했다. 특히 현장에서 쓰는 OS가 기존 목록에 없으면 기능이 없는 것처럼 보일 수 있어서, 지원 OS 기준을 코드와 설정에서 같이 맞췄다.

## 정리

리눅스 서버 비밀번호 변경 기능은 원격 명령 실행 기능처럼 보이지만, 실제로는 계정 권한 모델과 OS별 차이를 다루는 작업이었다.

비밀번호 변경 기능을 안정적으로 만들려면 접속, 권한, 명령, 결과 판정, 내부 상태 동기화를 각각 나눠서 검증해야 한다.
