# 비밀번호 변경 실패 이력이 남지 않던 문제

## 문제

장치에 적용되지 않는 비밀번호 규칙으로 변경을 시도했을 때, 변경은 실패했지만 실패 이력이 남지 않는 문제가 있었다.

성공 이력만 남으면 운영자는 "변경을 시도한 적이 없는 것"처럼 보게 된다. 실제로는 규칙 때문에 실패했는데, 이력이 없으면 원인 분석이 어려워진다.

## 내가 확인한 것

비밀번호 변경 흐름을 따라가면서 이력 저장 위치를 확인했다.

- 변경 요청이 들어오는 시점
- 비밀번호 규칙 검증이 실패하는 시점
- 장치로 실제 변경 요청을 보내는 시점
- 성공 후 이력을 저장하는 시점
- 예외가 발생했을 때 이력 저장을 건너뛰는지

문제는 일부 실패가 이력 저장 전에 반환되거나 예외 처리에서 빠지는 것이었다.

## 수정 방향

수정 방향은 실패도 감사 이벤트로 남기는 것이었다.

```text
요청 수신 -> 규칙 검증 -> 실패 시 실패 이력 저장 -> 사용자에게 실패 사유 표시
```

단, 모든 입력 오류를 이력으로 남기는 것은 과할 수 있다. 그래서 실제 변경 요청의 일부로 볼 수 있는 실패와, 화면 입력 단계에서 바로 막히는 오류를 구분했다.

## 결과 메시지

실패 이력을 남길 때는 메시지도 중요했다.

"실패"만 남기면 원인을 알 수 없다. 그래서 규칙 불일치, 장치 응답 실패, 내부 예외처럼 가능한 범위를 나눠 결과 메시지에 남기도록 했다.

## 정리

비밀번호 변경 기능에서 이력은 성공 기록만이 아니다.

실패 이력이 있어야 운영자가 어떤 규칙 때문에 실패했는지, 장치까지 요청이 갔는지, 내부에서 막혔는지 판단할 수 있다. 감사 관점에서는 실패도 중요한 이벤트다.
