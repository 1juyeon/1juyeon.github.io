# 보안 심사 대응용 의존성 업그레이드는 어디서부터 봐야 할까

## 상황

보안 심사나 인증 대응을 하다 보면 사용 중인 런타임, WAS, 프레임워크, DB, 네이티브 라이브러리의 지원 종료 여부를 확인해야 하는 순간이 온다.

이때 단순히 "전부 최신 버전으로 올리자"라고 접근하면 위험하다. 버전 하나를 올리는 일이 아니라, 런타임과 프레임워크, 서블릿 스펙, 설정 파일, 서드파티 라이브러리 호환성을 함께 옮기는 일이기 때문이다.

## 먼저 해야 할 일

가장 먼저 현재 사용 중인 구성요소를 표로 만든다.

| 구분 | 현재 버전 | 지원 상태 | 영향 범위 |
| --- | --- | --- | --- |
| JDK | 확인 필요 | LTS 여부 확인 | 컴파일, 런타임 |
| WAS | 확인 필요 | EOL 여부 확인 | Servlet/JSP 스펙 |
| Framework | 확인 필요 | EOL 여부 확인 | 전체 웹 계층 |
| DB | 확인 필요 | 메이저 버전 지원 기간 확인 | JDBC, dump/restore |
| Native Library | 확인 필요 | CVE, ABI 확인 | DLL/SO 연동 |

이 표를 만들면 "무엇부터 올려야 하는지"보다 "무엇이 서로 묶여 있는지"가 보인다.

## 업그레이드 순서

실무에서는 아래 순서가 비교적 안전하다.

1. JDK와 빌드 타깃을 먼저 정한다.
2. WAS가 지원하는 Servlet/Jakarta 스펙을 확인한다.
3. Spring 같은 웹 프레임워크의 요구사항을 확인한다.
4. 보안 프레임워크 설정 변경을 따로 분리한다.
5. JDBC 드라이버와 DB 서버 버전 호환성을 확인한다.
6. 네이티브 라이브러리는 ABI와 배포 파일 목록을 따로 검증한다.
7. 통합 테스트는 인증, 세션, 파일 업로드, 배치, 외부 연동부터 본다.

중요한 점은 WAS와 프레임워크를 따로 고르면 안 된다는 것이다. 예를 들어 프레임워크가 Jakarta 기반이면 WAS도 Jakarta 스펙을 지원해야 한다.

## "최신"보다 "호환되는 최신"

보안 심사 대응에서는 최신 버전이라는 말보다 "현재 코드와 호환 가능한 지원 버전"이 더 중요하다.

무작정 최신으로 올리면 다음 문제가 생길 수 있다.

- `javax.*`에서 `jakarta.*`로 전면 변경 필요
- XML schema namespace 변경
- Spring Security 설정 방식 변경
- 오래된 MyBatis, Jasypt, DBCP 같은 라이브러리 호환성 문제
- JSP taglib 또는 servlet-api 중복 충돌
- 빌드 산출물의 JDK class version 불일치

그래서 후보를 두 갈래로 나누는 것이 좋다.

| 경로 | 장점 | 단점 |
| --- | --- | --- |
| 최소 변경 경로 | 빠르고 회귀 위험이 낮음 | 다음 EOL이 빠를 수 있음 |
| 장기 지원 경로 | 오래 유지 가능 | 마이그레이션 범위가 큼 |

심사 일정이 촉박하면 최소 변경 경로를 검토하고, 장기 유지보수가 목표라면 큰 마이그레이션을 별도 일정으로 잡는 편이 안전하다.

## 테스트 우선순위

전체 화면을 다 누르기 전에 깨지면 치명적인 흐름부터 확인한다.

- 로그인과 세션 유지
- 권한별 메뉴 접근
- 비밀번호 변경과 암호화 저장
- DB 연결과 트랜잭션
- 스케줄러와 배치
- 외부 장비 또는 외부 SDK 연동
- 파일 업로드/다운로드

특히 보안 프레임워크를 올리면 CSRF, 세션 정책, password encoder, filter chain 순서가 바뀔 수 있다. 화면은 떠도 인증/인가가 미묘하게 달라질 수 있으니 별도 체크리스트로 봐야 한다.

## 정리

보안 심사 대응 업그레이드는 "버전 숫자 바꾸기"가 아니라 호환성 계약을 다시 맞추는 작업이다.

현재 버전 표를 만들고, EOL 여부를 확인하고, WAS와 프레임워크의 스펙 조합을 먼저 고른 뒤, 보안 설정과 외부 연동을 우선 테스트하는 흐름으로 가야 한다.
