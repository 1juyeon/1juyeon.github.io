# 외부 동기화에서 기존 값이 덮어써지는 문제

## 문제

외부 시스템에서 데이터를 다시 불러올 때, 이미 등록된 항목의 일부 정보가 덮어써지는 문제가 있었다.

특히 장치 계정 정보와 제조사, 모델명, 펌웨어, 프로토콜 같은 메타 정보는 성격이 다르다. 계정 정보는 최신값으로 갱신해야 할 수 있지만, 현장에서 직접 보정한 메타 정보까지 무조건 덮어쓰면 운영 데이터가 흐트러진다.

## 이번에 한 작업

장치 동기화 흐름에서 어떤 값은 업데이트하고, 어떤 값은 유지해야 하는지 기준을 다시 잡았다.

내가 수정한 방향은 다음과 같았다.

- 기존 장치의 아이디와 비밀번호는 필요 시 갱신
- 제조사, 모델명, 펌웨어, 프로토콜은 무조건 덮어쓰지 않도록 방어
- 장치 계정 갱신 시 다른 사용자 테이블에 중복 등록되지 않도록 수정
- 외부 연동 응답이 실패처럼 보여도 실제 장치 변경 결과를 별도로 판단
- 비밀번호 변경 인터페이스 변경 사항을 호출부에 반영
- 장치 비밀번호 조회 인터페이스를 추가해 확인 흐름 보강

## 왜 분리해야 했는가

동기화 기능은 보통 "외부 값이 정답"이라고 생각하기 쉽다.

하지만 운영 중에는 내부 시스템에서 보정한 값이 더 정확할 때도 있다. 예를 들어 자동 수집된 모델명은 비어 있거나 축약될 수 있고, 현장에서 확인한 값은 관리자가 직접 수정했을 수 있다.

그래서 동기화 대상 필드를 아래처럼 나눠 봤다.

| 필드 성격 | 처리 기준 |
| --- | --- |
| 접속 계정 정보 | 외부 시스템 기준으로 갱신 가능 |
| 장치 식별 정보 | 기존 값과 충돌 시 신중하게 처리 |
| 운영자가 보정한 메타 정보 | 무조건 덮어쓰기 방지 |
| 변경 결과 상태 | 응답 코드와 실제 결과를 분리해서 판단 |

## 응답 실패와 실제 변경

비밀번호 변경 작업에서는 더 애매한 케이스가 있었다.

장치 쪽 비밀번호 변경은 성공했지만, 외부 시스템 응답이 실패처럼 돌아오면 내부 시스템에는 반영되지 않는 문제가 생길 수 있다. 이 경우 다음 실행부터 내부에 저장된 값과 실제 장치 값이 달라진다.

그래서 응답만 보고 실패로 단정하지 않고, 가능한 경우 실제 변경 여부를 확인하는 흐름을 같이 봐야 했다.

## 정리

장치 동기화는 단순 덮어쓰기 작업이 아니다.

외부에서 가져온 값, 내부에서 관리자가 보정한 값, 실제 장치의 현재 상태가 서로 다를 수 있다. 그래서 필드별 갱신 기준과 실패 응답 처리 기준을 분리해야 데이터 무결성을 지킬 수 있다.
