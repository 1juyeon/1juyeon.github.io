# KEK와 DEK를 나눌 때 운영에서 꼭 확인할 것

## 상황

보안 요구사항이 올라가면 저장된 비밀번호나 자격증명을 단순 고정 키로 암호화하는 방식에서 벗어나야 할 때가 있다.

이때 자주 나오는 구조가 KEK와 DEK를 나누는 방식이다.

- DEK: 실제 데이터를 암호화하는 키
- KEK: DEK를 감싸거나 풀기 위한 상위 키

구조 자체는 좋지만, 운영 관점에서 확인할 것이 많다. 특히 KEK가 서버 기동 시 입력되는 마스터 비밀번호에서 유도되는 방식이라면 더 조심해야 한다.

## 장점

KEK/DEK 분리는 다음 장점이 있다.

- 데이터 암호화 키를 직접 노출하지 않을 수 있다.
- 마스터 키 교체와 데이터 재암호화를 분리할 수 있다.
- 기동 시 언락 전에는 민감 데이터를 복호화할 수 없게 만들 수 있다.
- 키 파기, 재생성, 접근 제어 정책을 더 명확히 만들 수 있다.

하지만 장점이 곧 운영 안정성을 보장하지는 않는다.

## 운영에서 생기는 문제

기동 시 마스터 비밀번호를 입력해야 KEK가 만들어지는 구조라면, 서버가 정상적으로 떠 있어도 애플리케이션은 아직 "언락되지 않은 상태"일 수 있다.

이 상태에서 민감 데이터를 복호화하려 하면 다음 문제가 생긴다.

- 저장된 DEK를 풀 수 없음
- 기존 암호문을 복호화하지 못함
- 외부 시스템 로그인에 사용할 비밀번호를 얻지 못함
- 화면에서는 단순 "로그인 실패"나 "연동 실패"처럼 보임

따라서 암호화 구조를 바꿀 때는 기능 코드만 수정하면 안 된다. 운영 상태 모델도 같이 설계해야 한다.

## 필요한 상태 구분

애플리케이션은 최소한 아래 상태를 구분할 수 있어야 한다.

| 상태 | 의미 |
| --- | --- |
| LOCKED | 서버는 떠 있지만 마스터 비밀번호가 입력되지 않음 |
| UNLOCKED | KEK가 준비되어 복호화 가능 |
| KEY_ERROR | 입력한 마스터 비밀번호가 기존 키와 맞지 않음 |
| MIGRATION_REQUIRED | 기존 암호문을 새 포맷으로 전환해야 함 |

이 구분 없이 복호화 실패를 모두 일반 오류로 처리하면 원인 추적이 어려워진다.

## 체크리스트

KEK/DEK 구조를 넣을 때는 다음을 확인한다.

- 서버 재시작 후 누가, 언제, 어떻게 언락하는가
- 언락 전 화면과 API는 어떤 메시지를 보여주는가
- 잘못된 마스터 비밀번호를 입력했을 때 기존 데이터가 손상되지 않는가
- 키가 메모리에만 있는지, 디스크에 남지 않는지
- 로그에 마스터 비밀번호나 파생 키가 찍히지 않는가
- 백업/복구 절차에서 키와 암호문이 함께 관리되는가
- 레거시 암호문을 새 포맷으로 옮기는 경로가 있는가

## 정리

KEK/DEK 분리는 보안상 좋은 방향이지만, 운영에서는 "서버가 떠 있음"과 "복호화 가능한 상태"가 달라진다.

그래서 암호화 로직 변경은 코드 변경이 아니라 운영 절차 변경까지 포함해야 한다. 언락 상태, 실패 메시지, 복구 절차, 마이그레이션 경로를 같이 설계해야 실제 장애를 줄일 수 있다.
