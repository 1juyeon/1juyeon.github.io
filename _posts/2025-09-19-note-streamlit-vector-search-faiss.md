## 이번에 한 작업

장애 대응 기록을 검색하고 분류하는 Streamlit 기반 AI 도구를 만들면서 벡터 검색 방식을 조정했다.

처음에는 로컬에서 잘 동작하는 벡터 DB 구성을 사용했다. 하지만 Streamlit Cloud 같은 배포 환경에서는 패키지 설치와 런타임 제약 때문에 같은 방식이 안정적으로 동작하지 않았다.

그래서 검색 구조를 더 단순한 파일 기반 FAISS 인덱스 방식으로 바꿨다.

## 내가 실제로 손댄 부분

- Streamlit 앱 화면 구성
- 장애/문의 기록을 읽어 분류하는 로직
- 벡터 검색용 문서와 메타데이터 구조
- 기존 벡터 DB 방식에서 FAISS 기반 검색으로 전환
- 배포 환경용 `requirements` 정리
- 디버깅 메시지 제거와 전체 구조 정리
- 간단한 분류 테스트와 fallback 테스트 추가

## 왜 바꿨는가

로컬에서 동작하는 구조가 배포 환경에서도 항상 맞지는 않았다.

벡터 DB는 기능이 많지만, 배포 환경에서 네이티브 의존성이나 저장소 권한 문제가 생기면 앱이 시작하지 못할 수 있다. 반면 FAISS 인덱스와 메타데이터 파일을 같이 두는 방식은 구조가 단순하고, 작은 규모의 검색 도구에는 충분했다.

```text
문서 로딩
-> 임베딩 생성
-> FAISS 인덱스 검색
-> 메타데이터 매칭
-> 관련 장애 유형 추천
```

## 검증하면서 본 것

검색 기능은 결과가 "그럴듯하게" 나오는 것만으로는 부족했다.

그래서 아래를 확인했다.

- 앱이 배포 환경에서 시작되는지
- 인덱스 파일이 없을 때 안내가 되는지
- 검색 결과와 메타데이터가 어긋나지 않는지
- 분류 실패 시 fallback이 동작하는지
- 디버깅 출력이 사용자 화면에 남지 않는지

## 정리

AI 도구를 만들 때는 모델 호출보다 배포 가능한 검색 구조가 더 먼저 막힐 때가 있다.

작은 내부 도구라면 복잡한 벡터 DB보다 단순한 인덱스 파일과 명확한 fallback이 더 안정적일 수 있다. 중요한 건 로컬 성공이 아니라 배포 환경에서 재현 가능하게 동작하는 구조를 만드는 것이다.
