# DB 엔진 업그레이드에서 백업 스크립트까지 같이 본 이유

## 이번에 한 작업

DB 엔진 메이저 업그레이드 작업을 하면서 애플리케이션 설정뿐 아니라 백업 스크립트까지 같이 확인했다.

DB 서버 버전만 바꾸면 끝나는 작업처럼 보이지만, 실제 운영에서는 JDBC 설정, 백업/복구 명령, 파일 경로, 인증 정보, 서비스 재기동 순서가 같이 묶인다.

내가 확인한 범위는 다음과 같았다.

- JDBC 접속 설정 수정
- 새 DB 버전에 맞는 드라이버와 연결 확인
- 백업 스크립트 경로 수정
- dump/restore 명령이 새 버전 기준으로 동작하는지 확인
- 애플리케이션 기동 후 주요 화면에서 DB 조회 확인
- 업그레이드 단계 문서화

## 백업 스크립트를 같이 본 이유

DB 업그레이드에서 가장 위험한 착각은 "애플리케이션이 DB에 붙으면 끝났다"라고 보는 것이다.

애플리케이션은 정상 기동되더라도, 나중에 백업이 실패하면 운영 리스크가 훨씬 커진다. 특히 DB 클라이언트 도구 경로가 바뀌거나, 백업 스크립트가 예전 설치 경로를 보고 있으면 장애는 즉시 드러나지 않는다.

그래서 업그레이드 검증에 백업 스크립트를 포함했다.

```text
DB 연결 확인 -> 주요 기능 확인 -> 백업 실행 확인 -> 복구 가능성 확인
```

## 설정 파일에서 본 점

JDBC 설정은 단순 URL 변경만 보지 않았다.

- 드라이버 클래스
- 접속 URL
- 인증 정보 암호화 여부
- 커넥션 풀 설정
- 백업 스크립트와 같은 값을 바라보는지

DB 접속 설정과 백업 스크립트가 서로 다른 값을 바라보면, 화면은 정상인데 백업만 실패하는 상태가 될 수 있다.

## 검증한 흐름

검증은 아래 순서로 했다.

1. 새 DB 버전으로 서비스 기동
2. JDBC 연결 확인
3. 로그인과 주요 조회 화면 확인
4. 백업 스크립트 실행
5. 백업 파일 생성 여부 확인
6. 실패 시 어느 경로와 명령에서 막혔는지 로그 확인

## 정리

DB 엔진 업그레이드는 애플리케이션 연결만 보면 부족하다.

운영 관점에서는 백업 스크립트, 복구 절차, 드라이버, 설정 파일이 모두 같은 버전을 바라봐야 한다. 그래서 DB 업그레이드를 할 때는 "붙는가"와 "운영 가능한가"를 따로 검증해야 한다.
