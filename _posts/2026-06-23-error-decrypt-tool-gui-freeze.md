## 이번에 한 작업

운영 설정 파일에 들어 있는 `ENC(...)` 값을 복호화할 수 있는 복구 도구를 만들었다.

처음에는 Java로 CLI와 Swing GUI를 만들었지만, 사용자 PC에 JDK가 없을 수 있고 실행도 번거로웠다. 그래서 최종적으로는 Go로 단일 `.exe`를 만드는 방향으로 바꿨다.

목표는 명확했다.

- 외부 라이브러리 없이 실행
- JRE/JDK 설치 없이 실행
- `ENC(...)` 줄을 그대로 붙여넣으면 괄호 안 암호문만 추출
- 복호화 버튼과 복사 버튼 제공
- 키 입력 UI는 두지 않음
- 복호화 알고리즘은 운영 애플리케이션과 동일하게 맞춤

Go에서 Jasypt를 직접 호출한 것이 아니라, 같은 포맷과 KDF, AES 복호화 과정을 표준 라이브러리로 재구현했다.

실제 구현은 이렇게 나눠서 했다.

- `ENC(...)` 전체 문자열을 붙여넣어도 내부 암호문만 꺼내는 파서 작성
- 운영 애플리케이션과 같은 KDF, salt 처리, AES 복호화 함수 작성
- CLI에서 먼저 known value 테스트가 통과하는지 확인
- Win32 창, 입력창, 복호화 버튼, 복사 버튼을 Go로 구성
- GUI와 CLI가 같은 복호화 함수를 공유하도록 구조 분리
- 배포용 단일 실행 파일로 빌드

## 처음 만든 뒤 생긴 문제

기능 자체는 테스트를 통과했다. 그런데 GUI에서 복호화 버튼을 여러 번 누르면 프로그램이 "응답 없음" 상태가 되는 문제가 있었다.

복호화 로직이 틀렸다면 매번 실패해야 하는데, 문제는 반복 클릭에서만 나타났다. 그래서 암호화 코드가 아니라 Win32 GUI 쪽을 봤다.

원인은 메시지 루프였다.

Go runtime은 goroutine을 여러 OS thread에서 스케줄링한다. 그런데 Win32 GUI는 창을 만든 thread와 메시지 루프를 도는 thread가 중요하다. 메시지 루프가 OS thread에 고정되어 있지 않으면 버튼 콜백과 메시지 처리가 꼬일 수 있다.

## 적용한 수정

GUI 루프 시작 지점에 OS thread 고정을 넣었다.

```go
func runGUI() {
    runtime.LockOSThread()
    defer runtime.UnlockOSThread()

    // create window
    // message loop
}
```

그리고 버튼 핸들러에 방어 코드를 넣었다.

- 복호화 중 panic이 나도 프로그램이 죽지 않게 `recover()` 추가
- 잘못된 입력은 상태 메시지로 표시
- 복사 버튼에서 클립보드 점유나 할당 실패가 나도 죽지 않게 처리
- GUI와 CLI가 같은 복호화 함수를 쓰도록 유지

## 검증한 방법

정확성과 안정성을 나눠서 검증했다.

복호화 정확성은 단위 테스트로 확인했다.

```text
known ENC sample -> expected plain text
```

GUI 안정성은 버튼 연타로 확인했다.

- 복호화 버튼 100회 연타
- 프로세스 생존 여부 확인
- 응답 없음 상태가 아닌지 확인
- 잘못된 입력에서도 상태 메시지를 표시하는지 확인

자동화로 GUI 결과 텍스트까지 읽으려 했지만, 다른 프로세스의 자식 컨트롤 텍스트를 단순 API로 읽는 데 제약이 있었다. 그래서 결과값 정확성은 함수 테스트로 고정하고, GUI는 멈춤 여부 중심으로 검증했다.

## 정리

이번 작업은 암호화 복구 도구를 만든 작업이면서 동시에 Win32 GUI를 Go로 만들 때의 thread 제약을 배운 작업이었다.

복호화 함수가 맞아도 GUI 메시지 루프가 OS thread에 고정되지 않으면 사용자가 버튼을 여러 번 누르는 현실적인 동작에서 멈출 수 있다. 단일 exe 도구를 만들 때는 알고리즘 검증과 UI 스트레스 검증을 따로 해야 한다.
