## 이번에 한 작업

운영 이력이 많아지면서 상세 화면 접근이 느려지는 문제를 정리했다.

이력은 감사와 장애 분석에 필요하지만, 무한히 쌓이면 조회 화면 성능을 떨어뜨린다. 특히 상세 정보를 볼 때 모든 이력을 가져오면 페이지 진입 자체가 느려질 수 있다.

내가 수정한 방향은 다음과 같았다.

- 비밀번호 변경 이력 데이터가 일정 개수를 넘지 않도록 주기적으로 정리
- 매일 새벽 정리 작업이 실행되도록 스케줄 추가
- 장치 상세 화면에서는 최신 이력 일부만 조회하도록 변경
- 이력 전체 보존과 화면 조회 성능을 분리해서 판단
- 관련 매퍼와 서비스 조회 기준 수정

## 왜 최신 일부만 조회했는가

장치 상세 화면에서 필요한 정보는 보통 최근 변경 상태다.

전체 이력이 필요할 때는 별도의 이력 화면이나 검색 조건이 있어야 한다. 상세 화면에서 모든 데이터를 한 번에 가져오면, 사용자는 단순히 장치 정보를 보려는 것뿐인데 오래 기다리게 된다.

그래서 상세 화면은 최신 이력만 보여주는 방향으로 정리했다.

```text
장치 상세 조회 -> 최신 이력 N건만 조회 -> 전체 분석은 별도 화면에서 처리
```

## 정리 작업 기준

이력 정리 작업은 단순 삭제가 아니라 보존 기준을 정해야 한다.

너무 많이 지우면 감사 자료가 사라지고, 너무 많이 남기면 성능 문제가 생긴다. 그래서 개수 기준을 두고, 일정 시각에 자동 정리되도록 했다.

## 정리

운영 이력은 남기는 것만큼 조회하는 방식도 중요하다.

상세 화면은 빠르게 보여야 하고, 감사 분석은 필요한 조건으로 조회할 수 있어야 한다. 이 둘을 같은 쿼리로 처리하면 데이터가 늘어날수록 화면 성능이 흔들린다.
