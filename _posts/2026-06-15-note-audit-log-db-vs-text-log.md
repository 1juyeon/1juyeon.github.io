# 보안 감사에서 말하는 로그는 텍스트 로그와 다르다

## 헷갈렸던 지점

보안 요구사항에서 "로그를 저장해야 한다"는 문구를 보면 처음에는 서버 로그 파일을 떠올리기 쉽다.

하지만 접근 이력, 로그인 실패, 계정 잠금, 중요 설정 변경 같은 보안 감사 항목에서 말하는 로그는 보통 텍스트 로그 파일만을 뜻하지 않는다.

운영 로그와 감사 기록은 목적이 다르다.

## 텍스트 로그

텍스트 로그는 개발자와 운영자가 장애를 분석하기 위해 보는 자료다.

예시는 다음과 같다.

```text
INFO user login success
WARN password change failed
ERROR external api timeout
```

장점은 남기기 쉽고 디버깅에 유용하다는 것이다. 하지만 감사 기록으로는 약점이 있다.

- 검색 UI가 없을 수 있다.
- 일반 관리자가 보기 어렵다.
- 보존 정책이 불명확할 수 있다.
- 로그 포맷이 코드 변경에 따라 흔들릴 수 있다.
- 누가 무엇을 언제 바꿨는지 구조적으로 조회하기 어렵다.

## DB 감사 기록

감사 기록은 구조화된 데이터로 남기는 편이 좋다.

예시는 다음과 같다.

| 컬럼 | 의미 |
| --- | --- |
| actor | 수행한 사용자 |
| action | 로그인, 로그아웃, 변경, 실패 등 |
| target | 대상 리소스 |
| result | 성공/실패 |
| reason | 실패 사유 |
| ip | 접근 위치 |
| created_at | 발생 시각 |

이렇게 남기면 날짜별 조회, 실패 횟수 집계, 특정 사용자 이력 확인, 관리자 전용 화면 제공이 가능하다.

## "저장됨"과 "점검 가능"은 다르다

감사 데이터가 DB에 쌓인다고 해서 요구사항을 만족했다고 보기 어렵다.

보안 점검에서 중요한 것은 관리자가 실제로 확인할 수 있느냐다.

- DB에는 쌓이지만 화면이 없음: 반쪽 구현
- 화면은 있지만 관리자 권한 제한이 없음: 보안 문제
- 실패 횟수는 있지만 차단 사유가 이력에 없음: 감사 추적 부족
- 날짜 검색은 있지만 다른 이력 테이블을 보여줌: 요구사항 불일치

즉 "로그 저장"과 "로그 점검 가능"은 별도 항목으로 봐야 한다.

## 구현 체크리스트

접근 로그나 감사 로그를 구현할 때는 다음을 확인한다.

- 로그인 성공, 실패, 로그아웃, 자동 로그아웃, 중복 로그인 차단이 구분되는가
- 로그인 실패 누적과 계정 잠금 시각이 남는가
- 중요 설정 변경 시 변경 전후 또는 변경 요약이 남는가
- 일반 사용자는 감사 기록 화면에 접근할 수 없는가
- 관리자 화면에서 날짜 범위 검색이 가능한가
- 엑셀 다운로드가 있다면 개인정보나 비밀값이 포함되지 않는가
- 텍스트 로그와 DB 감사 기록의 역할이 문서에 구분되어 있는가

## 정리

보안 감사에서 필요한 로그는 단순 `logger.info()` 출력이 아니라, 조회 가능하고 권한 제어되는 구조화된 감사 기록이다.

텍스트 로그는 장애 분석용, DB 감사 기록은 보안 추적용으로 역할을 나누고 설계해야 한다.
