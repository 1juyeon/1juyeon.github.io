비밀번호 변경 이력을 조회할 때 결과 코드 `1010`이 반복해서 보였고, 바로 뒤에 다음 로그가 남았다.

```text
change result: 1010
Cannot find this err[1010] in enumerate
```

처음에는 이 문장이 비밀번호 변경 실패의 원인이라고 생각했다. 코드를 따라가 보니 두 문제를 분리해야 했다.

```text
원래 문제: 네이티브 계층이 왜 1010을 반환했는가
추가 문제: Java 계층이 1010을 메시지로 바꾸지 못함
```

## 결과 코드가 올라오는 경로

```text
C/C++ 네이티브 모듈
  숫자 결과 코드 반환
        ↓ JNI
Java 서비스
  결과 이력 저장
        ↓
Java enum
  숫자를 화면 메시지로 변환
```

Java enum에는 주변 코드인 `1009`, `1015` 등이 있었지만 `1010`은 정의돼 있지 않았다. 따라서 이력 조회 자체는 저장된 숫자를 읽었지만, 화면에 표시할 문자열을 찾는 단계에서 별도 오류가 난 것이다.

## 숫자 순서로 의미를 추측하지 않았다

`1010`이 `1009` 다음에 있다는 이유로 비슷한 비밀번호 규칙 오류라고 넣을 수도 있었다. 하지만 숫자가 연속이라는 사실은 의미를 보장하지 않는다.

먼저 확인해야 할 자료는 다음 순서였다.

```text
1. 실제 배포 DLL과 같은 버전의 C/C++ 헤더
2. 네이티브 enum 또는 #define
3. SDK 문서와 릴리스 노트
4. 코드가 1010을 반환하는 분기
5. 같은 시각의 원본 통신 로그
```

이 자료로 의미가 확인되기 전까지 1010을 임의의 문구에 연결하지 않았다. 이 조사에서 확정한 것은 “1010의 원인은 이것이다”가 아니라 “Java enum이 네이티브 계약을 따라가지 못하고 있다”는 점이었다.

## 모르는 코드를 처리하는 방식

알 수 없는 외부 코드를 성공으로 넘기면 안 된다. 그렇다고 화면 전체를 예외로 깨뜨릴 필요도 없다.

```java
public ErrorView resolve(int code) {
    ErrorView known = ERROR_MAP.get(code);
    if (known != null) {
        return known;
    }

    log.warn("Unmapped native result code: {}", code);
    return new ErrorView(code, "정의되지 않은 외부 오류");
}
```

이렇게 하면 원본 숫자는 보존하면서 사용자 화면은 안전하게 실패로 표시할 수 있다. 이후 코드 의미가 확인되면 enum, 다국어 메시지, 테스트 케이스를 같은 변경에서 추가해야 한다.

## 정리

`Cannot find this err[1010] in enumerate`는 1010이 발생한 원인을 설명하는 로그가 아니다. 네이티브 모듈은 이미 1010을 반환했고, Java가 그 숫자를 해석하지 못했다는 두 번째 오류다.

이 문제를 보며 숫자 코드의 발생 지점과 표시 지점을 따로 추적했다. 그리고 외부 DLL을 바꿀 때는 함수 시그니처뿐 아니라 오류 코드 목록도 Java와 맞는지 확인해야 한다는 기준을 남겼다.
