# 로컬 AI 세션 아카이버에 Codex 로그와 이미지 보존을 추가한 기록

## 이번에 한 작업

로컬에 남는 AI 코딩 세션을 나중에 다시 찾아볼 수 있도록, 세션 아카이버에 Codex 로그를 읽는 source adapter를 추가했다. 기존에는 한 종류의 세션 로그만 다뤘는데, 도구마다 저장 형식이 달라도 같은 아카이브 구조로 모으는 것이 목표였다.

이번 변경에서는 대화 텍스트만 옮기는 데서 끝내지 않고, 세션 중간에 첨부한 이미지가 대화 흐름 안에서 다시 보이도록 저장하는 부분까지 손봤다.

## 로그 형식을 공통 세션 모델로 정리한 방법

Codex 로그는 JSONL 형식이라 한 줄씩 읽어야 한다. adapter에서는 로그 파일을 찾아 다음 정보로 정규화했다.

- 세션 식별자와 제목
- 생성·갱신 시각
- 사용자 메시지와 응답 메시지
- 명령 실행, 패치 적용 같은 도구 이벤트
- 이미지 첨부 정보

제목은 세션 색인에 있으면 먼저 사용하고, 없으면 세션 안의 제목 변경 이벤트나 첫 사용자 메시지로 대체했다. 이 순서를 두지 않으면 모든 파일이 비슷한 이름으로 쌓여 나중에 찾기 어려웠다.

JSONL 한 줄이 깨진 경우에는 조용히 넘기지 않고 파일과 줄 번호를 포함해 실패하도록 했다. 기록을 일부만 보관한 채 정상으로 보이는 것보다, 원본 상태를 먼저 확인하는 편이 안전하다고 판단했다.

## 도구 출력은 기본적으로 요약만 남겼다

세션 로그에서 가장 위험한 부분은 명령과 도구의 실제 출력이었다. 진단 로그나 환경 정보가 길게 붙을 수 있고, 의도하지 않은 값이 함께 남을 가능성도 있다.

그래서 기본 저장값은 `명령 완료`, `패치 적용 완료`처럼 실행 사실과 종료 상태만 남기도록 했다. 실제 출력을 보관해야 하는 경우에만 별도 옵션을 켜도록 분리했다. 메시지를 정규화하는 단계와 Markdown을 렌더링하는 단계 모두에서 민감한 값 치환을 적용해, 한 단계만 통과한 값이 결과물에 남지 않도록 했다.

## 이미지는 대화 순서를 유지해 별도 파일로 저장했다

이미지를 본문 끝에 한꺼번에 붙이면, 어떤 설명과 함께 입력한 이미지였는지 알기 어려웠다. 그래서 메시지를 읽을 때 이미지 자리에 내부 토큰을 넣고, Markdown을 쓸 때 실제 이미지 링크로 바꾸는 흐름으로 만들었다.

이미지 데이터는 별도 assets 디렉터리에 저장하고, Markdown에는 상대 경로 링크만 남겼다. 이 방식이면 본문 파일이 과도하게 커지지 않고 대화 중간의 이미지 위치도 유지된다.

실패 처리도 넣었다.

- 지원하지 않는 첨부 형식은 사용할 수 없다고 표시
- 손상된 base64 데이터나 없는 원본 파일은 해당 이미지만 건너뜀
- 세션 파일명이 바뀌어 다시 생성될 때는 이전 Markdown과 assets를 함께 정리
- 정리 대상이 예상한 세션 디렉터리 아래에 있는지 확인한 뒤에만 삭제

마지막 항목은 단순한 파일 정리 코드라도 경로 검증 없이 재귀 삭제를 호출하면 위험할 수 있어서 추가했다.

## 검증한 내용

테스트에서는 비밀번호 형태의 문자열이 치환되는지, 명령 이벤트가 기본값에서 요약으로만 남는지, 이미지가 본문 전후 문장 사이에 들어가는지를 확인했다. 이미지 바이너리가 정규화 JSON에 그대로 남지 않는 것도 함께 검사했다.

실제 내보내기 전에는 dry-run으로 발견 가능한 세션 수와 대상만 먼저 확인하고, 그 다음에 Markdown·색인·첨부 파일이 함께 만들어지는지 확인했다. 이 순서를 두니 저장 형식이 다른 두 종류의 로그를 추가해도 아카이브 쪽의 구조를 크게 바꾸지 않아도 됐다.

## 정리

이번 작업에서 중요했던 점은 AI 세션을 많이 저장하는 것보다, 나중에 다시 읽을 수 있는 형태와 보관 범위를 먼저 정한 것이었다. 메시지, 도구 이벤트, 이미지의 역할을 나누고 기본값을 보수적으로 두니, 아카이브는 유지하면서도 불필요한 원본 출력의 노출 가능성을 줄일 수 있었다.
