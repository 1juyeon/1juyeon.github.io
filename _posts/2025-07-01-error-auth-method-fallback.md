# 장비마다 다른 인증 방식 때문에 변경이 실패하던 문제

## 문제

같은 기능인데 어떤 장비에서는 인증이 통과하고, 어떤 장비에서는 인증 단계에서 실패하는 문제가 있었다.

처음에는 계정 정보가 틀렸거나 네트워크가 막힌 것처럼 보였다. 하지만 같은 계정 형식과 같은 요청 흐름에서도 장비별로 결과가 달랐다. 이 경우 원인은 비밀번호 변경 로직보다 인증 방식 차이에 있을 가능성이 컸다.

## 핵심 요약

| 항목 | 내용 |
| --- | --- |
| 내가 한 일 | 장비별 실패를 인증 방식, TLS 단계, 계정 오류로 나눠 진단했다. |
| 수정 방향 | 하나의 인증 방식에 고정하지 않고, 서버가 제시한 방식과 안전한 우선순위를 기준으로 fallback 흐름을 설계했다. |
| 로그 기준 | 민감한 값은 남기지 않고 시도한 인증 방식, 응답 코드, TLS 실패 여부만 남기도록 정리했다. |
| 남긴 결과 | 새 장비가 추가되어도 "어떤 인증 조건에서 실패했는가"로 판단할 수 있는 기준을 만들었다. |

## 내가 확인한 것

먼저 실패를 한 문장으로 묶지 않고 단계별로 나눴다.

- TCP 연결이 되는지
- TLS handshake가 통과하는지
- HTTP 401 응답에서 어떤 인증 방식을 요구하는지
- Basic, Digest, WS-Security 같은 인증 방식 중 무엇을 받아들이는지
- 같은 장비군에서도 펌웨어나 설정에 따라 요구 방식이 달라지는지
- 실패 로그가 "인증 실패"와 "통신 실패"를 구분하는지

인증 실패를 모두 같은 메시지로 보여주면 운영자는 원인을 알 수 없다. 그래서 어떤 인증 방식을 시도했고 어디서 실패했는지 로그에 남기는 것이 중요했다.

## curl로 인증 방식을 먼저 확인했다

애플리케이션에서 바로 확인하면 내부 로직과 외부 통신 문제가 섞인다.

그래서 curl로 서버가 요구하는 인증 방식을 먼저 확인했다.

```bash
curl -v -k "https://<device-host>/<api-path>"
```

여기서 `WWW-Authenticate` 헤더를 보면 서버가 요구하는 인증 방식을 알 수 있다.

```http
WWW-Authenticate: Digest ...
```

또는

```http
WWW-Authenticate: Basic ...
```

이 값을 기준으로 애플리케이션이 어떤 인증 옵션을 써야 하는지 좁혔다.

## 수정 방향

수정은 "하나의 인증 방식만 고정"하는 대신, 가능한 인증 방식을 순서대로 시도하고 결과를 남기는 방향으로 잡았다.

```text
연결 확인
-> 서버가 요구하는 인증 방식 확인
-> 우선순위에 따라 인증 시도
-> 실패 방식과 응답 코드 기록
-> 다음 방식 시도 또는 사용자에게 원인 표시
```

무조건 모든 방식을 계속 시도하면 장애가 더 복잡해질 수 있다. 그래서 인증 방식마다 실패 조건과 중단 조건을 나눴다.

| 상황 | 처리 |
| --- | --- |
| TLS handshake 실패 | HTTP 인증을 시도하지 않음 |
| 서버가 Digest만 제시 | Digest 우선 시도 |
| 서버가 여러 방식 제시 | 안전한 방식부터 시도 |
| 인증 정보 자체가 틀림 | fallback으로 숨기지 않고 실패 표시 |
| 지원하지 않는 방식 | 지원 범위 밖으로 명확히 기록 |

## 내가 신경 쓴 부분

fallback은 편하지만 위험할 수 있다.

인증 정보가 틀린 문제까지 fallback으로 덮으면, 사용자는 "언젠가 되겠지"처럼 보이는 느린 실패를 겪는다. 그래서 fallback은 인증 방식 차이를 흡수하는 용도로만 두고, 계정 오류나 TLS 오류는 별도로 드러나게 했다.

로그에는 다음 정보를 남기도록 정리했다.

- 시도한 인증 방식
- 서버가 제시한 인증 방식
- HTTP 상태 코드
- TLS 단계 실패 여부
- 최종 실패 원인

민감한 값은 남기지 않고, 방식과 결과만 남기는 것이 기준이었다.

## 검증한 케이스

검증은 장비 종류보다 인증 조건 중심으로 했다.

1. Basic만 허용하는 대상
2. Digest만 허용하는 대상
3. 여러 인증 방식을 제시하는 대상
4. TLS 인증서 검증에서 먼저 실패하는 대상
5. 계정 정보가 틀린 대상
6. 지원하지 않는 인증 방식을 요구하는 대상

이렇게 나누면 새 장비가 추가되어도 "어느 장비냐"보다 "어떤 인증 조건이냐"로 판단할 수 있다.

## 정리

장비 연동 실패는 비밀번호 변경 로직만의 문제가 아닐 때가 많다.

같은 API처럼 보여도 장비마다 인증 방식, TLS 설정, 오류 응답이 다를 수 있다. 그래서 인증 방식 확인, fallback 기준, 실패 로그를 분리해야 한다.

내가 한 일은 특정 장비 예외를 하나 더 추가한 것이 아니라, 인증 방식 차이를 진단 가능한 구조로 바꾼 것이다.
