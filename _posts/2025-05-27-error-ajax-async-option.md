DB 백업 파일을 업로드하고 복원 결과 이력을 보여주는 관리 화면을 만들었다. 일반적인 파일 업로드와 달랐던 점은 복원 과정에서 애플리케이션 서버 연결이 끊기거나 재시작될 수 있다는 것이었다.

이 때문에 서버에서는 복원이 진행됐는데 브라우저에서는 Ajax 오류가 보이거나, 결과를 받기 전에 페이지가 이동하는 문제가 생겼다.

## 구현한 화면 흐름

당시 화면에서는 파일을 선택하면 `FormData`로 복원 요청을 보내고, 결과에 따라 안내창을 띄운 뒤 목록을 다시 불러왔다.

```javascript
const formData = new FormData();
formData.append('file', selectedFile);

$.ajax({
  type: 'POST',
  url: '/admin/database/restore',
  data: formData,
  processData: false,
  contentType: false,
  success: showRestoreResult,
  error: showConnectionError
});
```

문제는 요청을 받은 서버 프로세스가 DB 복원과 함께 재시작되는 구조였다는 점이다. 서버가 작업을 끝냈더라도 HTTP 응답을 보내기 전에 연결이 끊기면 브라우저는 `status == 0`인 네트워크 오류로 볼 수 있다.

## `async: false`를 사용했던 이유와 한계

레거시 화면에서 후속 코드가 먼저 실행되지 않도록 요청을 동기로 바꿔 확인했다.

```javascript
$.ajax({
  async: false,
  // ...
});
```

동기 요청은 응답을 기다리는 동안 다음 JavaScript 실행을 막기 때문에 타이밍이 단순해졌다. 당시 구현에서는 복원 요청과 화면 새로고침의 순서를 고정하는 데 사용했다.

하지만 이것이 서버 재시작 문제를 해결한 것은 아니다.

- 요청 동안 브라우저 UI가 멈춘다.
- 서버가 응답 전에 내려가면 여전히 실패한다.
- 작업이 오래 걸리면 사용자는 멈춘 화면만 보게 된다.
- 최신 브라우저에서는 동기 XHR 사용이 권장되지 않는다.

즉 `async: false`는 작업 순서를 강제로 맞춘 임시 선택이었고, 장기적으로 유지할 구조는 아니었다.

## 결과를 세 가지로 나눠야 했다

복원처럼 서버 상태를 바꾸는 작업에서는 HTTP 성공/실패만으로 결과를 정하기 어렵다.

| 브라우저가 본 결과 | 실제 가능성 |
| --- | --- |
| 200 응답 | 복원 완료 또는 요청 접수 완료 |
| 500 응답 | 복원 실패 |
| 연결 끊김 | 서버 재시작, 네트워크 오류, 복원 도중 실패 중 하나 |

연결이 끊긴 경우에는 실패 안내만 띄우기보다 서버가 다시 올라온 뒤 복원 이력을 조회해 실제 결과를 확인하는 편이 맞다.

## 다시 설계한다면

복원 요청을 처리하는 프로세스가 스스로 재시작해야 한다면, 한 HTTP 연결로 끝까지 기다리는 설계 자체가 불안정하다.

```text
1. 복원 요청을 등록하고 작업 ID 반환
2. 별도 작업 프로세스가 복원 수행
3. 웹 애플리케이션 재기동
4. 브라우저가 health check로 재접속 확인
5. 작업 ID로 성공/실패 이력 조회
```

API는 다음처럼 “완료”가 아니라 “접수”를 반환할 수 있다.

```http
HTTP/1.1 202 Accepted
Content-Type: application/json

{"jobId":"restore-001","status":"accepted"}
```

다만 당시 코드에는 별도 작업 프로세스가 없었기 때문에, 화면에서는 동기 요청으로 순서를 고정하고 복원 결과 이력 화면을 함께 만들어 운영자가 결과를 다시 확인할 수 있게 했다.

## 정리

이 작업에서 구현한 것은 단순한 Ajax 옵션 변경이 아니었다.

- 백업 파일 업로드와 복원 요청 화면을 만들었다.
- 복원 성공/실패 이력을 조회할 수 있게 했다.
- 서버 재시작과 Ajax 응답이 충돌하는 타이밍을 확인했다.
- 동기 XHR은 순서를 맞추는 임시 대응일 뿐이라는 한계를 남겼다.

서버를 재시작하는 작업은 요청의 성공 여부와 실제 작업 결과를 분리해서 기록해야 한다. 다음에 같은 기능을 만든다면 비동기 작업 ID와 재접속 후 상태 조회를 먼저 설계할 것이다.
