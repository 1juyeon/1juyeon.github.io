# 오래된 라이브러리 업그레이드를 단계별로 나눠 진행한 기록

## 이번에 한 작업

오래된 Java 웹 프로젝트의 주요 라이브러리를 단계별로 업그레이드했다.

한 번에 모든 것을 올리면 빌드 오류와 런타임 오류가 섞여 원인을 찾기 어려워진다. 그래서 보안 로그 라이브러리, JDBC 드라이버, 네이티브 라이브러리 로더를 먼저 나눠서 진행했다.

내가 실제로 나눈 단계는 다음과 같았다.

- 오래된 로깅 라이브러리를 새 메이저 버전으로 전환
- JDBC 드라이버 업데이트
- 네이티브 SSL 라이브러리 로더가 특정 파일명에만 묶이지 않게 수정
- 빌드 오류를 단계별로 수정
- 실행 시 초기화 로그와 DB 연결을 확인
- 업그레이드 기록 문서 작성

## 왜 단계로 나눴는가

라이브러리 업그레이드는 버전 숫자만 바꾸는 일이 아니다.

로깅 라이브러리를 올리면 설정 파일 형식과 초기화 방식이 바뀔 수 있고, JDBC 드라이버를 올리면 DB 연결 옵션이나 드라이버 클래스 로딩 방식이 달라질 수 있다. 네이티브 라이브러리는 파일명과 로딩 경로가 맞지 않으면 런타임에서 바로 실패한다.

그래서 아래처럼 독립적으로 검증 가능한 단위로 나눴다.

```text
로깅 전환 -> 빌드 확인 -> 기동 로그 확인
JDBC 전환 -> DB 연결 확인 -> 주요 쿼리 확인
네이티브 로더 수정 -> 라이브러리 로딩 확인 -> 외부 통신 확인
```

## 네이티브 라이브러리 로더에서 본 문제

특정 버전 파일명에 로더가 강하게 묶여 있으면, 라이브러리 파일을 교체할 때마다 코드도 같이 바꿔야 한다.

그래서 로더가 버전이 붙은 파일명 하나만 찾는 구조인지 확인하고, 배포 파일명이 조금 달라져도 대응할 수 있게 정리했다.

이 부분은 빌드보다 배포 안정성과 더 관련이 있었다. 개발 환경에서는 파일이 있으니 잘 되지만, 설치본이나 패치본에서 파일명이 다르면 바로 장애가 날 수 있기 때문이다.

## 검증한 것

업그레이드 후에는 아래를 확인했다.

- 애플리케이션이 기동되는지
- 로깅 설정이 정상 적용되는지
- DB 연결이 성공하는지
- 주요 화면에서 쿼리 오류가 없는지
- 네이티브 라이브러리가 로딩되는지
- 기존 배포 구조에서 파일 누락이 없는지

## 정리

의존성 업그레이드는 큰 커밋 하나보다 작은 단계 여러 개가 더 안전했다.

각 단계가 어떤 문제를 해결하는지, 어떤 검증을 통과해야 다음 단계로 넘어갈지 정해두면 나중에 실패했을 때 되돌아갈 지점도 명확해진다.
