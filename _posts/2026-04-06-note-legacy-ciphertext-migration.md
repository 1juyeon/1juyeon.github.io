# 레거시 암호문을 새 암호화 방식으로 옮길 때의 함정

## 상황

기존 시스템의 비밀번호 저장 방식이 오래되어 새 암호화 방식으로 바꾸는 일이 있다.

이때 신규 저장 데이터만 보면 구현은 단순하다. 문제는 이미 DB에 들어 있는 기존 암호문이다.

기존 암호문을 새 복호화 함수로 읽으려 하면 당연히 실패한다. 그래서 보통 레거시 폴백을 둔다.

```text
새 포맷이면 새 복호화
아니면 기존 방식으로 복호화
```

하지만 "새 포맷인지 아닌지"를 어떻게 판별하느냐가 핵심이다.

## 위험한 방식: 휴리스틱 판별

암호문에 명확한 버전 정보가 없으면 길이, Base64 형태, 패딩 성공 여부 같은 단서로 포맷을 추측하게 된다.

이 방식은 위험하다.

- 레거시 암호문이 우연히 새 포맷처럼 보일 수 있다.
- 새 복호화가 예외 없이 이상한 평문을 만들 수 있다.
- 실패가 즉시 드러나지 않고 외부 로그인 실패 같은 후속 오류로 나타날 수 있다.
- 특정 데이터에서만 실패해 재현이 어렵다.

암호화 마이그레이션에서는 "복호화가 예외 없이 끝났다"와 "올바른 평문이다"를 구분해야 한다.

## 좋은 방식: 버전 prefix

가장 단순하고 강한 방식은 암호문에 버전을 붙이는 것이다.

```text
v1:<legacy-ciphertext>
v2:<new-ciphertext>
```

또는 JSON 구조로 감싸도 된다.

```json
{
  "version": 2,
  "alg": "AES-GCM",
  "ciphertext": "..."
}
```

이렇게 하면 복호화 라우팅이 추측이 아니라 명시적 계약이 된다.

## 마이그레이션 전략

실무에서는 한 번에 모두 바꾸기보다 단계적으로 가는 편이 안전하다.

1. 새 암호화 함수와 레거시 복호화 함수를 모두 둔다.
2. 읽을 때 포맷 버전으로 분기한다.
3. 레거시 값을 정상 복호화하면 저장 시 새 포맷으로 다시 저장한다.
4. 일정 기간 후 레거시 값이 얼마나 남았는지 집계한다.
5. 레거시 폴백 제거 시점을 별도 릴리즈로 잡는다.

이 과정을 기록하지 않으면 "어느 데이터가 어떤 암호화 방식인지" 운영자가 알 수 없다.

## 검증할 것

마이그레이션 테스트는 다음 케이스를 포함해야 한다.

- 새 포맷 신규 저장 후 복호화
- 레거시 포맷 기존 데이터 복호화
- 레거시 포맷을 새 포맷으로 재저장
- 잘못된 암호문 입력 시 명확한 오류 반환
- 키가 없거나 틀렸을 때 데이터가 덮어써지지 않음
- 외부 로그인 같은 후속 기능이 실제 평문으로 정상 동작

## 정리

암호화 방식 변경에서 가장 위험한 것은 새 알고리즘 자체보다 기존 데이터와의 경계다.

레거시 암호문을 추측으로 판별하지 말고, 가능하면 버전 prefix를 붙여 명시적으로 분기해야 한다. 그리고 마이그레이션은 읽기, 재저장, 관찰, 제거 단계를 나눠 진행하는 것이 안전하다.
