# 랜덤 비밀번호 생성에 DRBG를 적용하며 확인한 것

## 이번에 한 작업

비밀번호 랜덤 생성 로직에 보안 난수 생성 방식을 적용했다.

기존 기능은 화면에서 조건을 선택하면 비밀번호를 자동으로 만들어주는 흐름이었다. 이 부분은 사용자가 직접 정하는 비밀번호보다 운영 편의성이 좋지만, 생성 방식이 약하면 보안 기능 전체의 신뢰도가 낮아진다.

내가 수정한 범위는 다음과 같았다.

- 비밀번호 생성 로직에서 보안 난수 생성기를 사용하도록 변경
- 길이, 문자 종류, 허용 문자 조건을 생성 전에 검증
- 초기 비밀번호 변경 화면에서 생성 조건을 사용자가 확인할 수 있게 보강
- 난수 생성기 초기화 실패 시 조용히 다른 방식으로 넘어가지 않도록 오류 처리
- 생성 결과가 정책을 만족하는지 다시 확인

## 구현 흐름

나는 비밀번호 생성 흐름을 아래처럼 나눴다.

1. 화면에서 정책 조건을 선택한다.
2. 서버에서 조건을 검증한다.
3. 보안 난수 생성기를 초기화한다.
4. 조건에 맞는 후보 문자를 만든다.
5. 후보 문자에서 난수 기반으로 비밀번호를 구성한다.
6. 생성된 비밀번호가 길이와 문자 조합 조건을 만족하는지 다시 검사한다.

중요한 건 6번이었다. 생성 알고리즘이 맞다고 생각해도, 조건 조합에 따라 특정 문자군이 빠질 수 있다. 그래서 생성 후 검증을 한 번 더 두는 편이 안전했다.

## 나중에 수정한 초기화 오류

작업 후 난수 생성기 초기화 쪽에서 오류 케이스를 다시 보완했다.

보안 기능에서 가장 애매한 처리는 "초기화에 실패했지만 다른 방식으로 대충 생성하는 것"이다. 사용자는 정상 생성된 비밀번호라고 믿지만, 내부적으로는 기대한 보안 수준이 아닐 수 있다.

그래서 초기화 실패는 명확히 실패로 다루고, 화면이나 로그에서 원인을 찾을 수 있게 했다.

## 화면에서 같이 보강한 점

초기 비밀번호 변경 화면에는 안내 문구와 조건 표시도 같이 넣었다.

비밀번호 생성 규칙이 화면에 보이지 않으면 사용자는 왜 특정 길이만 가능한지, 왜 특정 문자가 들어가는지 알 수 없다. 그래서 생성 버튼만 두지 않고 현재 적용되는 조건을 같이 보여주는 방향으로 정리했다.

## 정리

랜덤 비밀번호 생성은 단순 문자열 조합이 아니라 난수 생성기, 정책 검증, 실패 처리, 화면 안내가 같이 맞아야 하는 기능이었다.

특히 보안 난수 적용은 코드 한 줄 교체로 끝나지 않는다. 초기화 실패를 어떻게 다룰지, 생성 결과가 정책을 만족하는지, 사용자가 어떤 조건으로 생성되는지 이해할 수 있는지까지 같이 봐야 한다.
