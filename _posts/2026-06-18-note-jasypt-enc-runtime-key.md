# 설정 암호화 알고리즘을 확인하고 복구 도구까지 맞춘 기록

## 이번에 한 작업

DB 비밀번호 변경 기능을 만들면서 설정 파일에 저장되는 `ENC(...)` 값이 정확히 어떤 방식으로 암호화되는지 확인했다.

단순히 "Jasypt를 쓴다"로는 부족했다. 나중에 복구 도구를 만들거나, 설정을 재생성해야 할 때는 알고리즘, provider, 키 위치, 포맷까지 알아야 하기 때문이다.

그래서 Spring 설정과 실제 서비스 코드를 따라가며 확인했다.

내가 확인하고 맞춘 구현 포인트는 아래였다.

- 설정 파일의 `ENC(...)` 값이 어느 placeholder를 통해 풀리는지 확인
- DataSource에 주입되는 시점과 DB 비밀번호 변경 서비스가 쓰는 encryptor가 같은지 확인
- 새 비밀번호 저장 시 기존 설정과 같은 알고리즘, provider, 키 기준을 쓰도록 맞춤
- 운영 복구 도구가 같은 결과를 내는지 known value 테스트로 고정
- 키 자체나 평문 비밀번호는 로그와 화면에 남지 않도록 확인

## 확인한 구조

설정 암호화는 다음 흐름이었다.

- 설정 파일에는 `ENC(...)` 형태로 저장
- Spring context가 기동될 때 encryptable property placeholder가 값을 복호화
- 복호화된 값이 DataSource 설정에 주입
- DB 비밀번호 변경 기능에서는 같은 encryptor 빈을 재사용해 새 값을 다시 `ENC(...)`로 저장

이렇게 확인하고 나니 DB 접속 설정과 변경 이력 저장이 같은 암호화 설정을 공유한다는 점도 같이 보였다.

## 내가 중요하게 본 점

암호문 자체보다 중요한 건 키 관리였다.

설정 파일에 암호문이 있어도 복호화 키가 없으면 애플리케이션은 DB에 붙지 못한다. 반대로 키를 아는 사람은 암호문을 평문으로 복원할 수 있다.

그래서 운영 문서에는 다음을 반드시 남겨야 한다고 정리했다.

- 복호화 키가 어디에서 주입되는지
- 서버 교체 시 키도 같이 이전해야 한다는 점
- 키가 없을 때 어떤 오류가 나는지
- 복구 도구가 애플리케이션 설정과 강하게 결합된다는 점

## 복구 도구와 연결

나중에 설정 파일의 `ENC(...)` 값을 복구하는 도구도 만들었다.

처음에는 Java로 직접 복호화 도구를 만들었고, 이후에는 단일 실행 파일이 필요해서 Go로 같은 알고리즘을 재구현하는 방향까지 검토했다.

이때 가장 중요한 기준은 "운영 애플리케이션과 같은 결과가 나오는가"였다.

그래서 복구 도구에는 known value 테스트를 두는 방식으로 정리했다.

```text
주어진 ENC 샘플 -> 기대 평문
```

설정 알고리즘이나 반복 횟수, 키 처리 방식이 바뀌면 이 테스트가 깨져야 한다.

## 정리

이번 작업에서 내가 한 일은 설정 값을 암호화해서 저장하는 기능만이 아니었다.

운영 애플리케이션이 쓰는 암호화 설정을 확인하고, DB 비밀번호 변경 기능과 복구 도구가 같은 기준을 쓰도록 맞췄다. 설정 암호화는 `ENC(...)` 문자열 하나가 아니라 암호문, 키, provider, 복구 도구, 테스트가 한 묶음으로 관리되어야 한다.
