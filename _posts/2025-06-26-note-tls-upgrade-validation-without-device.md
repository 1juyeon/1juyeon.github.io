# 테스트 장비 없이 TLS 라이브러리 업그레이드를 검증한 방법

## 상황

TLS 통신에 쓰는 라이브러리를 교체해야 했는데, 바로 테스트할 실제 장비가 없는 상황이 있었다.

라이브러리 업그레이드는 빌드가 되는 것만으로 충분하지 않다. 실제 통신에서 TLS handshake, 인증서 검증, HTTP 인증 방식, cipher 협상이 이전과 다르게 동작할 수 있기 때문이다.

그래서 나는 물리 장비가 없어도 통신의 핵심 조건을 나눠서 검증하는 방식으로 접근했다.

## 핵심 요약

| 항목 | 내용 |
| --- | --- |
| 내가 한 일 | 장비 고유 동작과 TLS 공통 동작을 분리해, 장비 없이도 확인 가능한 검증 범위를 만들었다. |
| 검증 도구 | curl, OpenSSL/libcurl 빌드 결과, 공개 테스트 서버, 자체 테스트 서버, 패킷 캡처를 함께 사용했다. |
| 확인한 증거 | TLS 버전, cipher 협상, 인증서 오류 위치, Digest 인증 도달 여부를 로그와 패킷으로 확인했다. |
| 남긴 결과 | 실제 장비가 없어도 업그레이드 위험을 줄일 수 있는 재사용 가능한 검증 절차를 남겼다. |

## 먼저 나눈 검증 범위

장비 자체를 그대로 재현할 수는 없지만, 통신을 구성하는 요소는 분리할 수 있다.

내가 나눈 검증 항목은 다음과 같았다.

- HTTPS handshake가 되는지
- 자체 서명 인증서 또는 신뢰되지 않은 인증서에서 어떤 오류가 나는지
- Digest 인증 요청/응답이 정상인지
- TLS 1.2 강제 같은 버전 제한이 적용되는지
- 기존 TLS 백엔드와 새 TLS 백엔드의 동작 차이가 있는지
- libcurl 빌드 결과가 의도한 SSL 백엔드를 사용하는지

이렇게 나누면 실제 장비가 없어도 "라이브러리 교체 자체가 통신 계층을 깨뜨렸는지"는 확인할 수 있다.

## 내가 만든 검증 흐름

검증은 curl 명령과 작은 테스트 서버를 조합해서 진행했다.

```text
libcurl/OpenSSL 빌드 확인
-> curl --version으로 SSL 백엔드 확인
-> 공개 HTTPS 테스트 서버로 handshake 확인
-> Digest 인증 테스트
-> TLS 버전 제한 테스트
-> 패킷 캡처로 실제 ClientHello 확인
```

중요한 건 "요청이 성공했다"에서 멈추지 않는 것이었다.

같은 요청이 성공하더라도 내부적으로 어떤 TLS 버전과 cipher를 쓰는지, 의도한 라이브러리로 링크됐는지 확인해야 했다.

## curl로 먼저 확인한 이유

애플리케이션에 바로 붙이면 문제가 났을 때 원인이 너무 많아진다.

- 애플리케이션 설정 문제
- JNI 또는 네이티브 로딩 문제
- 인증 정보 전달 문제
- TLS 라이브러리 문제
- 대상 서버 문제

그래서 먼저 curl 단독으로 통신을 확인했다.

```bash
curl -v --tlsv1.2 --digest -u "<user>:<password>" "https://<test-host>/<path>"
```

curl 단독 테스트가 통과하면, 최소한 TLS 백엔드와 인증 방식 자체는 동작한다고 볼 수 있다. 이후 애플리케이션에 붙였을 때 실패하면 범위를 JNI, 설정, 파라미터 전달 쪽으로 좁힐 수 있다.

## 패킷 캡처에서 본 것

라이브러리 업그레이드 검증에서는 로그만으로 부족할 때가 있다.

나는 패킷 캡처로 실제 TLS handshake를 확인했다.

- ClientHello가 나가는지
- 제안하는 TLS 버전이 기대와 맞는지
- cipher 목록이 달라졌는지
- 서버 인증서 오류가 TLS 단계에서 나는지
- HTTP 인증까지 도달하는지

TLS 단계에서 끊기면 HTTP 로그는 남지 않는다. 반대로 HTTP 401이 보이면 TLS는 이미 통과한 것이다. 이 구분만 해도 원인 범위가 크게 줄어든다.

## 검증 결과를 문서로 남긴 이유

물리 장비가 없는 상태에서 한 검증은 나중에 "정말 충분했나?"라는 질문을 받기 쉽다.

그래서 나는 성공 여부만 적지 않고, 어떤 조건을 어떤 방식으로 대체 검증했는지 남겼다.

| 검증 항목 | 대체 검증 방법 |
| --- | --- |
| TLS handshake | 공개 HTTPS 서버와 자체 테스트 서버 |
| 인증서 오류 | 자체 서명 인증서 또는 검증 실패 케이스 |
| Digest 인증 | 공개 Digest 테스트 엔드포인트 |
| TLS 버전 제한 | curl 옵션과 패킷 캡처 |
| 라이브러리 링크 | `curl --version`, DLL 로딩 경로 확인 |

이렇게 남겨두면 실제 장비 테스트가 나중에 가능해졌을 때도 어떤 항목을 추가로 보면 되는지 이어갈 수 있다.

## 정리

테스트 장비가 없다고 검증을 포기할 필요는 없다.

장비 고유 동작은 검증할 수 없더라도, TLS handshake, 인증서, Digest 인증, 라이브러리 링크, 패킷 레벨 동작은 충분히 분리해서 확인할 수 있다.

내가 한 일은 "장비 없이도 가능한 검증 범위"와 "나중에 실제 장비로 다시 봐야 하는 범위"를 나눈 것이다. 이 구분이 있어야 라이브러리 업그레이드를 감으로 배포하지 않을 수 있다.
