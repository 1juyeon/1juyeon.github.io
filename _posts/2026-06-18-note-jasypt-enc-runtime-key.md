# jdbc.properties의 ENC 값은 키 관리까지 같이 봐야 한다

## 상황

Java 애플리케이션에서 DB 접속 비밀번호를 `jdbc.properties`에 평문으로 두지 않고 `ENC(...)` 형태로 저장하는 경우가 있다.

보기에는 안전해 보이지만, 실제로는 "어떤 알고리즘으로 암호화했는가"보다 "복호화 키를 어떻게 관리하는가"가 더 중요하다.

## 확인할 것

Jasypt 같은 설정 암호화 도구를 쓴다면 다음을 확인해야 한다.

- 암호화 알고리즘
- 반복 횟수
- salt 생성 방식
- provider 의존성
- 복호화 키를 가져오는 위치
- 운영 서버에 키가 설정되어 있는지
- 키가 없을 때 애플리케이션이 어떻게 실패하는지

`ENC(...)` 문자열만 백업해도 키가 없으면 복구할 수 없다. 반대로 키를 아는 사람은 해당 값을 복호화할 수 있다.

## 환경변수 키의 장단점

복호화 키를 OS 환경변수로 두면 설정 파일과 키를 분리할 수 있다.

장점은 다음과 같다.

- 설정 파일이 유출되어도 키 없이는 복호화가 어렵다.
- 서버별로 키를 다르게 둘 수 있다.
- 코드에 키를 직접 넣지 않을 수 있다.

하지만 운영상 주의할 점도 있다.

- 환경변수가 빠지면 애플리케이션 기동 또는 DB 연결이 실패한다.
- 서버 교체 시 키 이전을 놓치기 쉽다.
- 복구 도구도 동일한 알고리즘과 키를 알아야 한다.
- 키 이름과 설정 위치를 문서화하지 않으면 장애 때 찾기 어렵다.

## 복구 도구와의 결합

운영 복구를 위해 `ENC(...)` 값을 평문으로 복원하는 도구를 만들 수 있다. 이때 도구는 애플리케이션의 암호화 설정과 강하게 결합된다.

즉 아래 값 중 하나라도 바뀌면 도구도 같이 바뀌어야 한다.

- 알고리즘
- 반복 횟수
- salt 처리
- 키 유도 방식
- provider 구현 차이

그래서 복구 도구에는 known value 테스트가 필요하다.

```text
주어진 ENC 샘플 -> 기대 평문
```

이 테스트가 깨지면 설정과 도구가 어긋난 것이다.

## 정리

`jdbc.password`에 들어가는 `ENC(...)` 값은 평문 저장보다 낫지만, 그 자체로 끝이 아니다.

암호문, 복호화 키, 알고리즘 설정, 복구 도구, 백업 절차가 한 묶음으로 관리되어야 한다. 특히 키가 환경변수에 있다면 서버 교체와 장애 복구 문서에 반드시 포함해야 한다.
