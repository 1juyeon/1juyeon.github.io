# 실제 장비 테스트 전 TLS/libcurl 업그레이드 위험을 줄인 방법

## 상황

HTTPS 통신에 쓰는 `curl/libcurl`과 OpenSSL 계열 라이브러리를 바꿔야 하는 작업이 있었다.

이 작업에서 제일 답답했던 부분은, 바로 붙여볼 실제 장비가 항상 준비되어 있지 않았다는 점이다. 장비가 없으면 당연히 장비 기능 검증은 할 수 없다. 특정 장비가 어떤 API를 지원하는지, 펌웨어마다 어떤 응답을 주는지, 실제 비밀번호 변경 같은 상태 변경이 되는지는 장비가 있어야 확인할 수 있다.

그래서 내가 한 일은 "장비 없이도 가능한 검증"과 "장비가 있어야만 가능한 검증"을 먼저 나누는 것이었다.

이 글은 장비 기능을 검증했다는 기록이 아니라, 실제 장비 테스트 전에 `TLS/libcurl` 교체로 생길 수 있는 공통 통신 계층 문제를 최대한 줄였던 기록이다.

## 먼저 정리한 기준

처음에는 "장비가 없으면 아무것도 못 하는 것 아닌가?"라는 생각이 들었다. 그런데 문제를 나눠보면 실제로는 장비와 상관없이 확인할 수 있는 부분도 있었다.

| 구분 | 장비 없이 확인 가능 여부 | 내가 확인한 방법 |
| --- | --- | --- |
| `libcurl`이 의도한 SSL 백엔드로 빌드됐는지 | 가능 | `curl --version`, DLL 로딩 경로 확인 |
| TLS 연결 자체가 되는지 | 가능 | 공개 HTTPS 서버, 자체 테스트 서버로 handshake 확인 |
| TLS 1.2 제한이 적용되는지 | 가능 | `--tlsv1.2` 옵션과 패킷 캡처 확인 |
| 인증서 오류가 어느 단계에서 나는지 | 가능 | 자체 서명 인증서, 신뢰되지 않은 인증서로 테스트 |
| Digest 인증 흐름까지 도달하는지 | 일부 가능 | Digest 테스트 엔드포인트에서 401 challenge 확인 |
| 장비 API가 실제로 원하는 동작을 하는지 | 불가능 | 실제 장비 필요 |
| 펌웨어별 응답 차이, 장비별 예외 동작 | 불가능 | 실제 장비 필요 |
| 비밀번호 변경처럼 상태가 바뀌는 기능 | 불가능 | 실제 장비 필요 |

여기서 중요한 건 "장비 없이 전부 검증했다"가 아니라, **장비가 없어도 먼저 줄일 수 있는 위험을 줄였다**는 점이다.

## 용어를 먼저 정리하면

`TLS`는 HTTPS에서 통신 내용을 암호화하기 위한 기술이다. 브라우저에서 `https://`로 접속할 때 내부적으로 서버와 암호화 방식을 협상하는데, 이 협상 과정을 보통 `TLS handshake`라고 부른다.

`cipher`는 암호화에 사용할 알고리즘 묶음이다. 같은 HTTPS 요청이라도 어떤 TLS 버전과 cipher를 쓰는지에 따라 오래된 서버나 장비에서 성공/실패가 갈릴 수 있다.

`libcurl`은 C/C++에서 HTTP 요청을 보낼 때 많이 쓰는 라이브러리다. Java 애플리케이션에서도 JNI 같은 방식으로 네이티브 DLL을 붙이면 내부 통신에 사용할 수 있다.

`SSL 백엔드`는 libcurl이 실제 암호화 처리를 맡기는 라이브러리다. Windows 기본 구성에서는 Schannel을 쓸 수 있고, 직접 빌드하면 OpenSSL을 쓰도록 만들 수 있다. 여기서 백엔드가 바뀌면 인증서 처리나 TLS 협상 결과가 달라질 수 있다.

`Digest 인증`은 HTTP 인증 방식 중 하나다. 비밀번호 원문을 바로 보내지 않고, 서버가 준 `nonce` 같은 값을 섞어서 해시를 만든 뒤 `Authorization` 헤더로 다시 보내는 방식이다. 그래서 Digest 인증에서는 처음 받은 `401 Unauthorized`가 최종 실패가 아니라, 서버가 "이 방식으로 인증해라"라고 알려주는 정상 흐름일 수도 있다.

## 내가 잡은 검증 순서

애플리케이션에 바로 붙여서 테스트하면 실패했을 때 원인이 너무 많아진다.

- DLL이 잘못 로딩됐는지
- 애플리케이션 설정이 잘못됐는지
- 계정/비밀번호 전달이 깨졌는지
- TLS 라이브러리 자체가 문제인지
- 서버 인증서 문제인지
- 장비 API가 다른 방식으로 동작하는지

그래서 먼저 애플리케이션 밖에서 `curl.exe` 단독 테스트를 진행했다.

```text
1. libcurl/OpenSSL 빌드 결과 확인
2. curl --version으로 실제 SSL 백엔드 확인
3. 공개 HTTPS 서버로 TLS handshake 확인
4. 자체 테스트 서버로 인증서 오류 케이스 확인
5. Digest 인증 테스트 엔드포인트로 401 challenge 확인
6. 패킷 캡처로 ClientHello와 TLS 버전 확인
7. 이후 애플리케이션에 붙였을 때의 문제 범위 좁히기
```

이 순서로 본 이유는 단순하다. 제일 아래쪽 통신 계층부터 확인해야 나중에 애플리케이션에서 실패했을 때 덜 헤맨다.

## 1. 빌드 결과부터 확인했다

라이브러리 업그레이드에서는 "빌드가 됐다"와 "내가 의도한 라이브러리가 실행되고 있다"가 다르다.

그래서 먼저 아래처럼 버전 정보를 확인했다.

```bash
curl.exe --version
```

확인하고 싶었던 건 이런 부분이었다.

```text
curl 8.x.x (Windows) libcurl/8.x.x OpenSSL/3.x.x
Protocols: http https ...
Features: SSL ...
```

여기서 `OpenSSL`이 보여야 내가 의도한 백엔드로 빌드된 것이다. 반대로 `Schannel`이 보이면 Windows 기본 TLS 백엔드를 쓰는 구성일 수 있다.

이 확인을 하지 않으면, 나중에 Digest 인증이 실패했을 때 "OpenSSL 기반으로 테스트했다"고 생각했지만 실제로는 다른 백엔드로 실행하고 있었던 상황이 생길 수 있다.

## 2. curl 단독으로 HTTPS 연결을 확인했다

다음은 실제 장비가 아닌 테스트 서버를 대상으로 HTTPS 연결 자체를 확인했다.

```bash
curl -vk --tlsv1.2 "https://<test-host>/"
```

여기서 본 것은 응답 본문보다 연결 과정이었다.

- TLS handshake가 시작되는지
- TLS 1.2 제한이 적용되는지
- 인증서 오류가 어느 시점에서 나는지
- HTTP 응답까지 도달하는지

`-v` 옵션은 요청/응답 과정을 자세히 보여준다. `-k`는 인증서 검증을 임시로 건너뛰는 옵션이다. 자체 서명 인증서나 테스트 서버에서는 필요할 수 있지만, 운영 환경에서 습관적으로 쓰면 안 된다.

이 단계에서 TLS가 실패하면 HTTP 인증까지 가지 못한다. 반대로 `HTTP/1.1 401 Unauthorized` 같은 응답을 받았다면, 최소한 TLS 연결은 지나갔다고 볼 수 있다.

이 구분이 생각보다 중요했다. 로그에 "인증 실패"처럼 보여도 실제로는 인증 전에 TLS에서 끊기는 경우가 있고, 반대로 401이 보이면 인증 단계까지는 도달한 것이다.

## 3. Digest 인증은 401을 먼저 봤다

Digest 인증은 보통 한 번에 성공하지 않는다.

흐름은 대략 아래와 같다.

```text
1. 클라이언트가 인증 정보 없이 요청
2. 서버가 401과 WWW-Authenticate: Digest ... 헤더 응답
3. 클라이언트가 nonce, realm, uri 등을 이용해 해시 계산
4. Authorization: Digest ... 헤더를 붙여 다시 요청
5. 서버가 해시를 검증하고 정상 응답 또는 실패 응답 반환
```

그래서 나는 먼저 서버가 Digest challenge를 제대로 주는지 확인했다.

```bash
curl -vk --digest -u "<user>:<password>" "https://<test-host>/<protected-path>"
```

이때 401이 무조건 나쁜 결과는 아니다. `WWW-Authenticate` 헤더 안에 `Digest`, `realm`, `nonce`, `qop`, `algorithm` 같은 값이 보이면 Digest 협상 흐름에 들어간 것이다.

내가 이 단계에서 보고 싶었던 건 "장비 API 성공"이 아니라 "TLS를 지나서 HTTP 인증 단계까지 도달하는가"였다.

## 4. 패킷 캡처로 로그를 보완했다

라이브러리 교체 작업은 로그만 보고 판단하기 어려울 때가 있었다.

예를 들어 로그에는 단순히 연결 실패로 남는데, 실제로는 ClientHello가 나가지 않았는지, 서버가 TLS 버전을 거절했는지, 인증서 검증에서 끊겼는지 구분이 안 될 수 있다.

그래서 패킷 캡처로 아래 항목을 확인했다.

| 확인 항목 | 본 이유 |
| --- | --- |
| ClientHello가 나가는지 | 클라이언트가 TLS 협상을 시작했는지 확인 |
| TLS 버전 | TLS 1.2 제한이 실제로 적용됐는지 확인 |
| cipher 목록 | 업그레이드 후 제안 목록이 달라졌는지 확인 |
| 서버 인증서 전달 여부 | 인증서 단계까지 도달했는지 확인 |
| HTTP 401 응답 여부 | TLS 이후 HTTP 인증 단계까지 갔는지 확인 |

`ClientHello`는 TLS 연결을 시작할 때 클라이언트가 서버에 보내는 첫 메시지다. 쉽게 말하면 "나는 이런 TLS 버전과 암호화 방식을 지원한다"라고 서버에 알려주는 단계다.

이걸 보면 단순한 성공/실패보다 훨씬 정확하게 원인을 나눌 수 있었다.

## 5. 애플리케이션에 붙일 때는 원인 후보를 줄인 상태로 봤다

curl 단독 테스트가 통과했다고 해서 애플리케이션 통합까지 성공한다는 뜻은 아니다.

다만 이 상태에서는 실패 원인을 더 좁힐 수 있다.

| 상황 | 원인 후보 |
| --- | --- |
| curl 단독 TLS 실패 | 빌드, SSL 백엔드, 인증서, TLS 버전 문제 가능성 |
| curl 단독 TLS 성공, Digest challenge 확인 | TLS 계층은 통과했고 HTTP 인증 단계까지 도달 |
| curl 단독 성공, 애플리케이션 실패 | JNI 로딩, DLL 경로, 옵션 전달, 계정 문자열 처리 문제 가능성 |
| 실제 장비에서만 실패 | 장비 API, 펌웨어, 인증 정책, 장비별 예외 가능성 |

이 기준이 없으면 나중에 실제 장비 테스트를 할 때도 계속 처음부터 다시 보게 된다. 나는 최소한 "이미 확인한 것"과 "아직 확인 못 한 것"을 분리해서 남기려고 했다.

## 내가 실제로 얻은 것

이 작업에서 제일 크게 남은 것은 명령어 몇 개가 아니라 검증 순서였다.

처음에는 장비가 없다는 사실 때문에 막힌 느낌이 컸다. 그런데 장비가 없어도 빌드 산출물, TLS 연결, 인증서 오류, Digest challenge, 패킷 레벨 동작은 확인할 수 있었다.

반대로 장비 API의 실제 응답, 펌웨어별 차이, 상태 변경 기능은 끝까지 "실제 장비 필요"로 남겼다. 이걸 억지로 검증했다고 쓰지 않는 것이 더 중요했다.

결국 내가 한 일은 다음과 같다.

- 장비가 없어도 확인 가능한 TLS/libcurl 공통 계층을 분리했다.
- `curl --version`과 DLL 로딩 기준으로 실제 사용 중인 SSL 백엔드를 확인했다.
- HTTPS 연결, 인증서 오류, Digest challenge 도달 여부를 curl로 먼저 확인했다.
- 패킷 캡처로 TLS handshake와 ClientHello를 확인했다.
- 실제 장비 테스트가 필요할 항목을 따로 남겨서 나중에 이어서 볼 수 있게 했다.

## 정리

이 작업은 "장비 없이 장비 기능을 검증했다"가 아니다.

정확히는 실제 장비 테스트 전에, 장비와 무관하게 확인 가능한 `TLS/libcurl` 위험을 먼저 줄인 작업이었다.

혼자 문제를 쪼개서 보면서 느낀 건, 테스트 환경이 부족할수록 더 구체적으로 나눠야 한다는 점이었다. 할 수 없는 검증은 할 수 없다고 남기고, 대신 할 수 있는 검증은 로그와 패킷으로 증거를 남겨야 다음 단계에서 덜 흔들린다.
