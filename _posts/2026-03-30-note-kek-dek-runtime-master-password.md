# 런타임 마스터 비밀번호 기반 KEK/DEK 구조로 바꾸며 정리한 것

## 이번에 한 작업

보안 심사 대응 과정에서 저장된 자격증명을 기존 고정 키 기반 복호화 흐름에서 더 분리된 키 구조로 바꾸는 작업을 했다.

내가 잡은 방향은 실제 데이터 암호화 키와 그 키를 보호하는 키를 나누는 구조였다.

- 실제 자격증명은 DEK로 암호화
- DEK는 KEK로 감싸서 저장
- KEK는 서버 기동 후 운영자가 입력한 마스터 비밀번호에서 유도
- 마스터 비밀번호와 KEK는 파일에 저장하지 않고 런타임 상태로만 유지

실제로 손댄 부분은 암호화 함수 하나가 아니었다.

- 서버 기동 직후에는 복호화가 막히도록 런타임 상태 값을 추가
- 운영자가 마스터 비밀번호를 입력했을 때만 KEK를 만들고 메모리에 유지
- 자격증명을 읽는 진입점에서 언락 여부를 먼저 확인
- 기존 암호문은 바로 덮어쓰지 않고 마이그레이션 대상으로 분류
- 키가 틀린 경우와 아직 키가 없는 경우의 오류 메시지를 분리

처음에는 "암호화 방식만 바꾸면 되는 일"처럼 보였는데, 구현하면서 보니 핵심은 암호화 알고리즘보다 운영 상태를 어떻게 표현하느냐였다.

## 구현하면서 바꾼 관점

기존에는 서버가 떠 있으면 곧바로 저장된 값을 복호화할 수 있는 구조였다.

하지만 KEK를 런타임 마스터 비밀번호에서 유도하면 서버가 떠 있어도 아직 복호화가 불가능한 상태가 생긴다.

그래서 상태를 이렇게 나눠서 생각했다.

| 상태 | 의미 |
| --- | --- |
| LOCKED | 서버는 기동됐지만 마스터 비밀번호가 아직 입력되지 않음 |
| UNLOCKED | KEK가 준비되어 저장된 자격증명을 복호화할 수 있음 |
| KEY_ERROR | 입력한 마스터 비밀번호가 기존 데이터와 맞지 않음 |
| MIGRATION_REQUIRED | 기존 암호문을 새 구조로 옮겨야 함 |

이 구분을 하지 않으면 사용자는 단순히 "외부 로그인 실패"나 "비밀번호 오류"처럼 보게 된다. 실제 원인은 언락이 안 됐거나 키가 맞지 않는 것일 수 있는데도 말이다.

## 내가 신경 쓴 부분

가장 조심한 부분은 복호화 실패가 데이터를 망가뜨리지 않게 하는 것이었다.

키가 없거나 틀렸을 때는 절대로 새 값으로 덮어쓰면 안 된다. 그래서 복호화 실패와 저장 실패를 분리해서 보려고 했다.

- 마스터 비밀번호가 없으면 복호화 시도 자체를 막기
- 키가 틀리면 "인증 실패"가 아니라 "키 불일치"로 다루기
- 암호문 포맷을 모르면 마이그레이션 대상으로 분류하기
- 실패한 값을 다시 암호화해 저장하지 않기
- 로그에 마스터 비밀번호, 파생 키, 평문 자격증명이 남지 않게 하기

이 작업을 하면서 "보안 기능"은 암호화 함수를 호출하는 코드가 아니라 실패 상태를 안전하게 다루는 코드라는 생각이 들었다.

## 운영에서 남긴 체크리스트

이 구조를 넣으면 배포 후 운영 절차도 바뀐다.

- 서버 재시작 후 누가 언락할 것인가
- 언락 전에는 어떤 화면과 메시지를 보여줄 것인가
- 잘못된 마스터 비밀번호 입력 시 재시도 정책은 어떻게 할 것인가
- 백업한 암호문과 키 관리 정보는 함께 복구 가능한가
- 장애 대응 문서에 "서버 기동"과 "암호화 언락"을 따로 적었는가

특히 외부 시스템 연동 비밀번호까지 이 체계를 타게 되면, 언락 상태가 곧 연동 가능 상태가 된다. 이걸 운영자가 모르면 서버는 정상인데 연동만 실패하는 애매한 장애로 보인다.

## 정리

이번 작업에서 내가 실제로 바꾼 것은 단순히 암호화 함수를 교체한 것이 아니라, 자격증명 복호화가 "런타임 언락 상태"에 의존하도록 만든 것이다.

그래서 앞으로 비슷한 작업을 할 때는 알고리즘 선택보다 먼저 상태 모델, 실패 메시지, 마이그레이션, 복구 절차를 같이 설계해야 한다.
