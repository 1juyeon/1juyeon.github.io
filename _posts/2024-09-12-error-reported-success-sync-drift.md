# 성공 로그 뒤에 숨어 있던 동기화 불일치 추적기

## 문제

운영 작업이 성공 또는 롤백된 것처럼 보이는데, 실제 외부 시스템의 상태는 기대와 다르게 남는 문제가 있었다.

겉으로는 단순했다. 애플리케이션 로그에는 변경 요청과 후속 처리가 남아 있었고, 실패 시 원복 흐름도 있었다. 그런데 실제 사용자는 일부 대상에서 접속이나 연동이 끊긴다고 했다. 로그만 믿으면 "성공했거나 원복됐으니 문제가 없어야 하는" 상황이었다.

이 문제는 기능 하나의 성공/실패가 아니라, 여러 시스템 사이에 같은 값이 유지되는지 확인하는 문제였다.

## 핵심 요약

| 항목 | 내용 |
| --- | --- |
| 내가 한 일 | 애플리케이션, 대상 시스템, 외부 연동 시스템의 최종 상태를 나눠 비교했다. |
| 확인한 경계 | Java 서비스 로직에서 네이티브 연동부까지 변경 전/후 값이 어떻게 전달되는지 추적했다. |
| 판단 기준 | 로그의 성공 여부가 아니라 세 시스템의 실제 상태 일치 여부를 기준으로 삼았다. |
| 남긴 결과 | 유사 장애에서 먼저 확인할 상태 대조 순서와 롤백 점검 기준을 문서화했다. |

## 내가 먼저 의심한 것

처음부터 특정 모듈을 원인으로 단정하지 않고 상태를 세 군데로 나눴다.

- 애플리케이션 DB에 저장된 값
- 실제 대상 장비 또는 계정에 적용된 값
- 외부 연동 시스템에 남아 있는 값

성공 로그가 있어도 세 곳의 값이 같다는 뜻은 아니다. 특히 변경 요청, 롤백 요청, 외부 동기화 요청이 서로 다른 API나 스레드에서 처리되면 한쪽만 다른 상태로 남을 수 있다.

그래서 로그 메시지보다 실제 상태 대조를 먼저 기준으로 잡았다.

```text
애플리케이션 상태
-> 대상 시스템 상태
-> 외부 연동 시스템 상태
-> 각 단계의 요청/응답 시각 비교
```

## 추적한 흐름

내가 확인한 핵심은 Java 서비스 코드에서 네이티브 연동부까지 값이 어떻게 흘러가는지였다.

- 변경 전 값과 변경 후 값이 어느 시점에 생성되는지
- 실패로 판단하는 기준이 응답 코드인지, 후처리 결과인지
- 롤백이 실제 대상에만 적용되는지, 외부 연동 시스템까지 같이 적용되는지
- 비동기 호출이 실패로 보고된 뒤에도 서버 쪽에는 반영될 수 있는지
- 같은 작업 ID로 각 시스템 로그를 연결할 수 있는지

문제는 "변경이 실패했다"가 아니라 "실패로 판단한 뒤 어느 시스템까지 원복했는가"에 가까웠다.

## 정리한 가설

가장 위험한 시나리오는 다음과 같았다.

```text
1. 애플리케이션이 변경 요청을 보낸다.
2. 대상 시스템 또는 외부 연동 시스템 중 일부에는 변경이 반영된다.
3. 호출 결과는 실패 또는 timeout으로 돌아온다.
4. 애플리케이션은 롤백을 수행한다.
5. 하지만 외부 연동 시스템 한쪽에는 새 값이 남는다.
```

이 경우 로그에는 실패 처리와 롤백이 남아 있어도, 실제 운영 상태는 "일부만 새 값"인 불일치 상태가 된다.

나는 이 현상을 조용한 불일치로 보고, 성공/실패 로그보다 시스템별 최종 상태를 기준으로 보고서를 정리했다.

## 해결 방향

이런 문제는 단순히 예외 처리를 하나 추가해서 끝낼 수 없다.

내가 정리한 방향은 다음과 같았다.

- 변경 단계마다 "요청 성공"과 "상태 일치"를 분리해서 기록
- 실패 시 롤백 대상 범위를 명확히 정의
- 외부 연동 시스템에 반영된 값도 별도 확인
- timeout이나 비동기 실패는 실제 반영 여부를 다시 조회
- 운영 점검용으로 최종 상태 대조 절차를 문서화

특히 중요했던 건 "실패 응답이면 반영되지 않았다"라고 가정하지 않는 것이다. 외부 시스템은 요청을 처리했지만 클라이언트가 실패로 받는 경우가 있다.

## 검증한 것

검증은 아래 관점으로 나눴다.

1. 정상 변경 시 세 시스템의 값이 모두 같은지
2. 중간 실패 시 어떤 시스템까지 변경됐는지
3. 롤백 후에도 외부 연동 시스템에 새 값이 남는지
4. timeout 발생 시 실제 반영 여부를 재조회할 수 있는지
5. 운영자가 로그만 보고 잘못 판단하지 않도록 점검 기준이 남는지

## 정리

이 작업에서 배운 점은 "성공 로그"가 곧 "운영 상태 일치"를 의미하지 않는다는 것이다.

여러 시스템을 동시에 바꾸는 기능에서는 요청 결과보다 최종 상태가 더 중요하다. 로그가 성공이라고 말해도 실제 대상과 외부 연동 시스템을 대조해야 하고, 실패처럼 보여도 서버에는 반영됐을 수 있다는 가능성을 열어두고 봐야 한다.

내가 한 일은 특정 오류 하나를 고친 것보다, 성공/실패 판단 기준을 실제 상태 검증 중심으로 다시 세운 것에 가까웠다.
