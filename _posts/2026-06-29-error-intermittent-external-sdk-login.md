# 외부 SDK 로그인이 간헐적으로 실패할 때 보는 순서

## 상황

외부 시스템에 SDK로 로그인하는 기능이 있다. 그런데 항상 실패하는 것이 아니라, 어떤 때는 되고 어떤 때는 안 된다.

이런 증상은 코드만 보면 오래 걸린다. "항상 실패"와 "간헐 실패"는 원인 후보가 다르기 때문이다.

## 먼저 증상을 분류한다

로그를 기준으로 실패 지점을 나눈다.

| 로그 양상 | 의미 |
| --- | --- |
| 연결 자체 실패 | 네트워크, 포트, 방화벽, 서버 상태 확인 |
| 연결은 성공, 인증 실패 | 계정, 세션 정책, 중복 로그인, 서버 측 거부 사유 확인 |
| 로그인 성공 후 목록 수신 실패 | SDK 내부 polling timeout, 서버 부하, 데이터량 확인 |
| 호출이 멈춘 듯 보임 | lock, thread 대기, native deadlock 확인 |

중요한 것은 "로그인 실패"라는 화면 메시지를 그대로 믿지 않는 것이다. 내부적으로는 연결 실패, 인증 거부, 후처리 타임아웃이 모두 같은 메시지로 보일 수 있다.

## 코드 변경이 원인인지 판단하는 기준

자격증명 복호화나 파라미터 매핑이 잘못됐다면 보통 매번 실패한다. 같은 입력이 항상 같은 잘못된 결과를 만들기 때문이다.

반대로 간헐적으로 성공한다면 아래 가능성이 더 커진다.

- 서버가 계정당 단일 세션만 허용
- 이전 세션 정리가 늦게 끝남
- 같은 계정을 다른 클라이언트가 사용 중
- 서버 부하로 로그인 후 데이터 수신이 timeout
- SDK 내부 전역 세션을 여러 thread가 공유
- 재시도 없이 한 번의 실패를 바로 사용자에게 노출

즉 간헐 실패는 코드 diff보다 런타임 상태와 서버 로그를 먼저 봐야 할 때가 많다.

## SDK 로그에서 볼 것

가능하면 SDK 로그에 다음 정보가 있어야 한다.

- thread id
- login start 시각
- connect 성공 여부
- login callback 성공 여부
- release/logout 호출 여부
- timeout까지 걸린 시간
- 서버가 내려준 실패 코드 또는 메시지

예를 들어 연결은 성공했는데 로그인 callback이 실패라면, 네트워크보다 서버의 인증 거부 사유를 봐야 한다. 이 거부 사유는 애플리케이션 로그가 아니라 외부 서버 로그에만 남는 경우가 많다.

## 동시성 여부 확인

외부 SDK가 내부에 전역 세션을 가진 구조라면 호출을 직렬화해야 할 수 있다.

```java
private static final ReentrantLock SDK_LOCK = new ReentrantLock(true);

public LoginResult login() {
    SDK_LOCK.lock();
    try {
        return sdk.login();
    } finally {
        SDK_LOCK.unlock();
    }
}
```

다만 lock을 넣었다고 끝은 아니다. `login`, `release`, `getList`, `changePassword`처럼 같은 세션을 건드리는 모든 경로가 같은 lock을 써야 한다.

그리고 lock을 잡은 뒤 예외가 나도 반드시 해제되어야 한다.

```java
try {
    // SDK 호출
} finally {
    releaseSafely();
}
```

## 서버 쪽 확인이 필요한 경우

애플리케이션에서는 "Login fail" 정도만 보이는데, 외부 서버는 더 정확한 이유를 알고 있을 수 있다.

- 중복 세션
- 최대 접속 수 초과
- 계정 잠금
- 비밀번호 만료
- 특정 IP 차단
- 서버 내부 세션 정리 지연

연결은 성공했지만 인증이 거부되는 경우라면 서버 로그가 결정적이다.

## 완화책

근본 원인을 확인한 뒤에는 다음을 검토할 수 있다.

- release 후 재로그인 사이에 짧은 대기 추가
- 로그인 실패 시 1~2회 제한적 재시도
- 계정 공유 제거
- SDK 호출 직렬화
- 서버 로그와 애플리케이션 로그의 trace id 또는 시각 동기화
- timeout 값을 데이터량과 서버 부하에 맞게 조정

재시도는 마지막에 넣는 편이 좋다. 원인 없이 재시도만 넣으면 실제 인증 문제를 숨길 수 있다.

## 정리

외부 SDK 로그인 간헐 실패는 "코드가 틀렸다"보다 "서버 상태, 세션 정책, SDK 전역 상태, timeout" 문제일 가능성이 높다.

먼저 실패 지점을 연결, 인증, 후처리, 대기로 나누고, 애플리케이션 로그와 외부 서버 로그를 같은 시각으로 맞춰 보는 것이 가장 빠르다.
