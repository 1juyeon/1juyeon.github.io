# DB 비밀번호 변경 기능에서 같이 바꿔야 하는 것들

## 상황

애플리케이션에서 DB 계정 비밀번호를 변경하는 기능을 만들 때, DB의 비밀번호만 바꾸면 끝이라고 생각하기 쉽다.

하지만 실제 운영에서는 DB 비밀번호가 여러 곳에 흩어져 있다.

- DB 사용자 비밀번호
- 애플리케이션의 DB 접속 설정
- 백업 스크립트의 인증 정보
- 변경 이력
- 복구용 설정 백업

하나라도 빠지면 DB 변경은 성공했는데 애플리케이션이 접속하지 못하거나, 백업만 조용히 실패하는 상황이 생긴다.

## 변경 흐름

안전한 변경 흐름은 대략 이렇게 나눌 수 있다.

1. 현재 설정과 필수 파일 존재 여부 확인
2. 새 비밀번호 생성 또는 입력값 검증
3. DB 계정 비밀번호 변경
4. 애플리케이션 설정 파일 갱신
5. 백업 스크립트 갱신
6. 변경 이력 저장
7. 재시작 또는 연결 재검증
8. 실패 시 가능한 범위에서 롤백

핵심은 "DB 변경"과 "애플리케이션 설정 반영" 사이에 장애가 나면 시스템이 깨질 수 있다는 점이다.

## 실패 지점을 나눠야 한다

비밀번호 변경 기능은 실패 위치에 따라 대응이 달라야 한다.

| 실패 위치 | 대응 |
| --- | --- |
| 입력 검증 전 | 이력 저장 없이 사용자에게 오류 |
| DB 변경 전 | 기존 상태 유지 |
| DB 변경 후 설정 파일 갱신 실패 | DB 비밀번호 롤백 또는 복구 절차 안내 |
| 설정 파일 갱신 후 백업 스크립트 실패 | 비밀번호 변경은 성공, 백업 설정 경고 |
| 재시작 실패 | 변경은 성공했지만 서비스 재시작 복구 필요 |

모든 실패를 같은 메시지로 처리하면 운영자가 어느 단계까지 반영됐는지 알 수 없다.

## 설정 파일 백업

DB 접속 설정 파일은 애플리케이션이 기동하는 데 필요한 핵심 파일이다. 변경 전후 백업을 남겨야 한다.

다만 백업 정책도 정해야 한다.

- 로컬에는 최신 1개만 남길지
- 외부 백업 폴더에 최신본만 덮어쓸지
- timestamp 이력을 누적할지
- 재패치나 재배포 때 살아남는 위치인지

재패치 과정에서 설정 파일이 사라질 수 있다면, 애플리케이션 폴더 바깥에도 최신 백업을 남기는 편이 안전하다.

## 화면 이력

비밀번호 변경 이력에는 최소한 다음 정보가 필요하다.

- 변경 시각
- 수행자
- 결과
- 실패 메시지

변경 전후 비밀번호를 보여줘야 하는 요구가 있다면 반드시 마스킹, 권한 제한, 복사 버튼 동작, 노출 시간 등을 따로 봐야 한다.

가능하면 평문 비밀번호 노출은 줄이고, 필요할 때만 관리자 권한으로 잠깐 확인하도록 만드는 것이 좋다.

## 정리

DB 비밀번호 변경 기능은 `ALTER USER` 한 줄로 끝나는 기능이 아니다.

DB, 애플리케이션 설정, 백업 스크립트, 이력, 백업 파일, 재시작까지 하나의 트랜잭션처럼 생각해야 한다. 완전한 DB 트랜잭션으로 묶을 수 없는 파일 작업이 섞이기 때문에, 단계별 실패 메시지와 복구 절차가 특히 중요하다.
