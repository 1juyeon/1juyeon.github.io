# 네이티브 DRBG 초기화 오류를 thread-local로 정리한 기록

## 문제

보안 난수 생성기를 네이티브 모듈에서 사용할 때 초기화 오류가 발생하는 케이스가 있었다.

난수 생성은 여러 요청 흐름에서 호출될 수 있고, 네이티브 라이브러리 안에서는 전역 상태나 초기화 순서가 꼬이면 재현하기 어려운 오류가 생긴다.

## 내가 확인한 것

처음에는 단순 초기화 누락으로 봤지만, 호출 흐름을 따라가 보니 스레드와 상태 관리가 더 중요했다.

확인한 항목은 다음과 같았다.

- 난수 생성기가 어느 시점에 초기화되는지
- 여러 스레드에서 같은 상태를 공유하는지
- 초기화 실패 후 재시도할 수 있는지
- Java/JNI 호출 경계에서 생성기 상태가 안전한지
- 실패 시 조용히 약한 생성 방식으로 넘어가지 않는지

## 수정 방향

수정은 생성기 상태를 스레드별로 분리하는 방향으로 잡았다.

```text
요청 스레드 진입
-> thread-local 생성기 확인
-> 없으면 초기화
-> 난수 생성
-> 실패 시 명확한 오류 처리
```

전역 객체 하나를 여러 스레드가 공유하면 초기화 순서나 내부 상태가 꼬일 수 있다. 그래서 각 스레드가 독립적인 생성기 상태를 갖도록 정리했다.

## 보안 기능에서 중요한 점

보안 난수 생성 실패는 대체 로직으로 조용히 넘어가면 안 된다.

사용자는 정상적으로 비밀번호가 생성됐다고 생각하지만, 실제로는 기대한 보안 수준이 아닐 수 있다. 그래서 초기화 실패는 명확히 실패로 드러나야 하고, 로그에서도 원인을 찾을 수 있어야 한다.

## 정리

네이티브 보안 모듈은 알고리즘뿐 아니라 초기화와 스레드 모델이 중요하다.

DRBG 같은 생성기는 호출 위치가 많아질수록 전역 상태보다 thread-local 상태가 더 안전할 수 있다. 보안 기능은 정상 결과뿐 아니라 초기화 실패와 재시도 흐름까지 같이 검증해야 한다.
