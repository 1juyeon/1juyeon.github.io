# 비밀번호 규칙 관리와 검증 전용 기능을 만들며 정리한 것

## 이번에 한 작업

비밀번호 생성 규칙을 관리하는 화면과, 실제 변경 없이 비밀번호가 규칙을 통과하는지만 확인하는 기능을 추가했다.

운영 화면에서는 비밀번호를 바로 변경하는 기능도 필요하지만, "이 규칙으로 생성하거나 입력한 값이 장치에 적용 가능한지" 먼저 확인하고 싶을 때가 있다. 그래서 검증 전용 흐름을 따로 두었다.

내가 수정한 부분은 다음과 같았다.

- 비밀번호 검증만 수행하는 서버 API 추가
- 장치 상세 화면에서 검증 기능을 사용할 수 있게 UI 추가
- 비밀번호 생성 규칙 목록에서 항목 선택 기능 추가
- 규칙 전체 삭제 기능 추가 후, 실제 사용 중인 규칙은 삭제되지 않도록 보완
- 최소/최대 길이 기준을 화면과 서버에서 맞춤
- 규칙 선택 UI에서 마우스 이벤트와 입력 이벤트 처리 방식 조정

## 검증 전용 기능을 둔 이유

비밀번호 변경은 실패했을 때 영향이 크다.

특히 외부 장치나 서버 계정에 실제로 변경 요청을 보내면, 실패 위치에 따라 내부 이력과 실제 장치 상태가 달라질 수 있다. 그래서 실제 변경 전에 규칙만 검증하는 기능이 필요했다.

```text
입력값 작성 -> 규칙 검증 -> 통과 여부 확인 -> 실제 변경 진행
```

이렇게 나누면 사용자는 변경 전에 실패 가능성을 줄일 수 있고, 운영자는 어떤 규칙 때문에 막히는지 먼저 알 수 있다.

## 삭제 기능에서 본 문제

규칙 관리 화면에는 선택 삭제와 전체 삭제가 있었다.

하지만 이미 장치나 계정에서 사용 중인 규칙까지 삭제되면, 나중에 이력을 보거나 변경 작업을 할 때 기준이 사라진다. 그래서 삭제 요청이 들어와도 사용 중인 규칙은 삭제하지 않도록 막았다.

이 부분은 단순 CRUD가 아니라 참조 무결성 문제였다.

## 정리

비밀번호 규칙 화면은 규칙을 추가하고 지우는 화면처럼 보이지만, 실제로는 운영 중인 비밀번호 변경 기능의 기준점을 관리하는 화면이다.

그래서 검증 전용 흐름, 길이 기준, 선택 삭제, 사용 중인 규칙 보호까지 같이 봐야 했다.
