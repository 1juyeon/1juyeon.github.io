# JSON 설정 파일이 있는데도 null로 로딩될 때

## 상황

장비나 외부 시스템별 동작을 JSON 템플릿으로 분리해두면 새 대상을 추가할 때 코드 수정 없이 설정만 추가할 수 있다.

그런데 파일은 분명히 배치했는데 런타임에서는 `jsonObj`가 `null`이고, 이후 단계에서 `NullPointerException`이 나는 경우가 있다.

이때 파일 존재 여부만 보면 원인을 놓칠 수 있다.

## 파일이 있어도 로딩되지 않는 이유

JSON 템플릿 로딩은 보통 아래 단계를 거친다.

1. 대상의 제조사, 모델, 타입 같은 식별 값을 읽는다.
2. DB 또는 설정 테이블에서 템플릿 이름을 찾는다.
3. 템플릿 이름으로 실제 파일 경로를 만든다.
4. 파일을 읽고 JSON 파서를 통과시킨다.
5. 파싱 결과를 실행 엔진에 넘긴다.

따라서 실제 파일이 있어도 2단계에서 템플릿 이름을 못 찾으면 파일은 읽히지 않는다.

## 체크 순서

가장 먼저 매핑 값이 만들어졌는지 확인한다.

- DB 또는 설정 테이블에 대상 row가 있는가
- 제조사, 모델, 타입 값이 정확히 일치하는가
- 템플릿 이름 컬럼이 비어 있지 않은가
- 확장자를 포함해야 하는지, 제외해야 하는지 규칙이 맞는가

그 다음 실제 배포 경로를 확인한다.

- 로컬 저장소가 아니라 운영 서버가 읽는 경로에 파일이 있는가
- 서비스 실행 계정에 읽기 권한이 있는가
- 파일 인코딩과 JSON 문법이 정상인가

마지막으로 스키마를 확인한다.

같은 `.json` 확장자라도 모든 엔진이 같은 구조를 읽는 것은 아니다. 어떤 엔진은 순번 키를 기대하고, 다른 엔진은 배열과 이름 있는 step을 기대할 수 있다. 파일 확장자가 같다고 호환된다고 보면 안 된다.

## 나쁜 실패 방식

JSON 로딩 실패를 내부에서 삼키고 `null`만 남겨두면, 나중에 전혀 다른 위치에서 예외가 난다.

```java
try {
    jsonObj = readTemplate(path);
} catch (Exception e) {
    log.warn("template load failed", e);
}

jsonObj.get("steps"); // 여기에서 NPE
```

이렇게 되면 화면에는 "통신 실패" 같은 일반 메시지만 보이고, 실제 원인은 템플릿 미등록 또는 로딩 실패였다는 사실이 숨겨진다.

## 개선 방식

로딩 성공 여부를 명시적으로 들고 가는 편이 좋다.

```java
Template template = templateLoader.load(templateName);

if (!template.loaded()) {
    return Result.fail("템플릿을 읽을 수 없습니다.");
}
```

또는 생성자에서 바로 실패시키고 상위 계층이 의미 있는 오류 코드를 반환하게 한다.

```java
if (templateName == null || templateName.isBlank()) {
    throw new MissingTemplateException("template name is empty");
}
```

핵심은 `null` 상태로 다음 단계에 진입하지 않는 것이다.

## 운영에서 자주 놓치는 부분

설정 파일 문제는 코드보다 배포 상태에서 많이 발생한다.

- 저장소에는 파일이 있지만 서버에는 없음
- 서버에는 파일이 있지만 DB 매핑 값이 없음
- DB 매핑 값은 있지만 템플릿 이름이 오타임
- 템플릿은 맞지만 실행 엔진이 다른 스키마를 기대함
- 서비스 계정 권한 때문에 파일을 못 읽음

정리하면, "파일이 있다"는 것은 JSON 로딩 조건 중 하나일 뿐이다. 매핑 값 생성, 실제 배포 경로, 파일 권한, JSON 스키마, 로딩 실패 처리까지 순서대로 봐야 한다.
