외부 관리 서버에 SDK로 로그인하는 기능이 간헐적으로 실패했다. 항상 실패하지 않고 같은 설정으로 성공과 실패가 번갈아 나왔기 때문에, 최근 코드 변경을 전부 원인 후보로 놓고 하나씩 대조했다.

이 조사의 결론은 특정 코드 한 줄을 원인으로 확정한 것이 아니다. 로그와 재현 조건으로 잘못된 가설을 지우고, 다음에 필요한 증거를 외부 서버 로그까지 좁힌 것이다.

## 확인된 증상

실패 시 SDK 로그는 다음 순서였다.

```text
TLS option: false
Launch login process
Login fail
```

이 로그로 알 수 있는 것은 다음과 같았다.

- TLS를 사용하지 않는 설정이었다.
- SDK 초기 구동과 서버 연결 단계는 지나갔다.
- 로그인 완료 callback이 실패로 돌아왔다.
- 실패는 매번이 아니라 간헐적이었다.

화면의 “로그인 실패” 한 문장보다 SDK 내부 단계가 더 중요했다.

## 1. SDK 바이너리 교체 가설

Git 이력상 특정 시점에 SDK DLL이 올라와 있어 처음에는 바이너리 교체를 의심했다. 하지만 실제 배포 환경에서는 그 바이너리를 이미 사용하고 있었고 Git 반영만 늦었다는 사실을 확인했다.

```text
Git 반영 시점 ≠ 실제 배포 시점
```

따라서 커밋 날짜만 근거로 “이 DLL 교체 후 문제가 생겼다”고 결론 내릴 수 없었다. 이 가설은 제외했다.

## 2. TLS 설정 불일치 가설

SDK에는 TLS 사용 여부를 전달하는 설정이 있었다. 서버와 값이 다르면 로그인 실패가 날 수 있어 확인했지만, 실제 운영 설정은 `false`였고 같은 설정으로 정상 동작하는 것도 확인됐다.

실패 로그에도 TLS 비활성화가 명확히 남아 있었기 때문에 이번 간헐 실패의 원인에서는 제외했다.

## 3. 자격증명 복호화 가설

비밀번호 암복호화 코드 변경도 후보였다. 그러나 동일한 암호문과 키로 복호화한 결과가 잘못됐다면 같은 입력에서 계속 잘못된 비밀번호가 나와야 한다.

```text
결정적인 복호화 오류 → 매번 실패할 가능성이 높음
현재 증상             → 같은 조건에서 간헐 실패
```

간헐성만으로 모든 암호화 문제를 배제할 수는 없지만, 고정된 복호화 로직 오류를 1순위로 보기에는 증상과 맞지 않았다.

## 4. 애플리케이션 동시성 가설

SDK가 전역 세션을 사용해 로그인, 목록 조회, 비밀번호 변경, release가 서로 충돌할 가능성이 있었다. 그래서 SDK 호출을 하나의 전역 lock으로 직렬화하고 `finally`에서 release가 보장되는 경로를 확인했다.

```java
SDK_LOCK.lock();
try {
    return sdk.loginAndLoad();
} finally {
    sdk.release();
    SDK_LOCK.unlock();
}
```

하지만 이 직렬화가 적용된 버전에서도 같은 현상이 발생했다. 애플리케이션 내부의 동시 호출만으로는 증상을 설명할 수 없어 이 가설도 제외했다.

## 로그가 가리킨 경계

남은 사실을 다시 정리했다.

```text
서버 연결 단계는 통과
TLS 경로 아님
고정 복호화 오류와 증상 불일치
애플리케이션 호출 직렬화 후에도 재현
로그인 callback은 서버 거부로 종료
```

애플리케이션과 SDK 로그에는 단순히 `Login fail`만 남고, 서버가 왜 거부했는지는 나오지 않았다. 이 시점부터는 클라이언트 코드를 더 추측하는 것보다 실패 시각의 서버 로그가 필요했다.

## 서버 쪽에서 확인해야 할 것

- 같은 계정을 다른 클라이언트가 사용 중인지
- 계정당 단일 세션 정책이 있는지
- 이전 logout 후 서버 세션 정리가 지연됐는지
- 최대 접속 수나 로그인 빈도 제한이 있는지
- 계정이 일시 잠금 상태였는지

특히 외부 클라이언트의 중복 로그인과 서버 내부 세션 정리는 애플리케이션의 lock으로 막을 수 없다.

## 확정 전에는 재시도를 넣지 않았다

짧은 지연 뒤 1~2회 재시도하면 증상 빈도는 줄 수 있다. 하지만 서버가 실제로 계정 잠금이나 잘못된 자격증명을 이유로 거부하는 상황까지 숨길 수 있다.

따라서 순서는 다음이 맞다고 판단했다.

```text
실패 시각 고정
→ 클라이언트/SDK 로그 연결
→ 서버 로그의 거부 사유 확인
→ 원인에 따라 세션 정리, 계정 분리 또는 제한적 재시도 적용
```

## 정리

이 조사는 원인을 확정하지 못한 채 끝난 것이 아니라, 현재 가진 증거로 확정할 수 없는 범위를 구분한 기록이다.

- Git 커밋 시점과 실제 배포 시점의 차이를 확인했다.
- TLS, 고정 복호화 오류, 애플리케이션 동시성 가설을 증상으로 배제했다.
- 연결 성공 뒤 로그인 callback에서 거부된다는 경계를 찾았다.
- 다음 증거가 외부 관리 서버의 로그인 거부 로그라는 데까지 좁혔다.

확실한 근거가 없을 때 가장 의심스러운 코드를 고치는 것보다, 이미 맞지 않는 가설을 지우고 결정권을 가진 서버의 로그를 요청하는 편이 안전했다.
