# 외부 SDK 초기화 실패를 원인 후보별로 쪼개 본 기록

## 문제

외부 SDK를 초기화하는 단계에서 실패했는데, 로그에는 넓은 의미의 "초기화 실패" 정도만 남는 경우가 있었다.

이런 오류는 바로 수정 코드를 쓰기 어렵다. 초기화 실패라는 말 안에 파일 로딩 실패, 권한 문제, 비트수 불일치, 네트워크 문제, 인증 문제, 서버 상태 문제가 모두 들어갈 수 있기 때문이다.

그래서 나는 원인을 하나씩 찍어보는 대신, 가능한 원인 후보를 분류하고 점검 순서를 만들었다.

## 왜 체크리스트가 필요했나

외부 SDK 문제는 애플리케이션 코드만 보면 원인을 놓치기 쉽다.

SDK는 보통 내부에서 다음을 함께 처리한다.

- DLL 또는 네이티브 라이브러리 로딩
- 런타임 의존성 확인
- COM 또는 레지스트리 등록 상태 확인
- 서버 연결
- 인증
- 세션 생성
- 장비 또는 채널 목록 조회

이 중 하나만 실패해도 애플리케이션에는 같은 오류 코드로 올라올 수 있다.

## 내가 나눈 원인 후보

먼저 원인을 네 범주로 나눴다.

| 범주 | 확인할 것 |
| --- | --- |
| 실행 환경 | OS 버전, 32/64비트, 서비스 계정 권한, 작업 디렉터리 |
| 파일/등록 상태 | DLL 존재 여부, 의존 DLL, COM 등록, 환경 변수 |
| 네트워크/서버 | IP/포트 접근, 방화벽, 서버 기동 상태, timeout |
| 인증/세션 | 계정 상태, 중복 로그인, 세션 제한, 권한 정책 |

이렇게 나누면 "초기화 실패"를 막연하게 보지 않고, 어느 범주에서 막혔는지 먼저 좁힐 수 있다.

## 점검 순서

내가 만든 점검 순서는 아래와 같았다.

```text
1. SDK 파일과 의존 DLL 존재 여부 확인
2. 애플리케이션과 SDK의 32/64비트 일치 확인
3. 서비스 실행 계정 권한 확인
4. SDK가 요구하는 등록 정보 확인
5. 서버 IP/포트 연결 확인
6. 인증 정보와 계정 상태 확인
7. 로그인 후 목록 조회까지 도달하는지 확인
8. 실패 시 SDK 로그와 애플리케이션 로그 시각 맞추기
```

이 순서를 둔 이유는 간단하다. 파일이 없거나 비트수가 다르면 네트워크를 볼 필요가 없고, 연결이 안 되면 인증을 볼 필요가 없다.

## 로그에서 분리한 기준

로그 메시지도 아래처럼 나눠서 보려고 했다.

- load 실패: SDK 파일 또는 의존성 문제
- init 실패: 환경, 등록 정보, 라이선스, 내부 초기화 문제
- connect 실패: 네트워크 또는 서버 상태 문제
- login 실패: 인증, 계정, 세션 정책 문제
- list 조회 실패: 로그인 이후 권한 또는 데이터 수신 문제

사용자에게는 모두 "연동 실패"처럼 보일 수 있지만, 개발자가 봐야 하는 지점은 완전히 다르다.

## 내가 실제로 정리한 것

코드 수정뿐 아니라 진단 기준을 같이 정리했다.

- 오류 코드가 만들어지는 위치 추적
- SDK 로그가 남는 경로 확인
- 서비스 계정과 수동 실행 계정 차이 확인
- timeout과 인증 실패를 같은 메시지로 묶지 않도록 분리
- 현상별로 먼저 볼 항목을 체크리스트화

중요했던 건 "가능한 원인을 많이 아는 것"보다 "어떤 순서로 지울 것인가"였다.

## 정리

외부 SDK 초기화 실패는 한 번에 맞히는 문제가 아니다.

초기화라는 단어 안에 실행 환경, 파일 의존성, 등록 상태, 네트워크, 인증, 세션 정책이 모두 들어갈 수 있다. 그래서 원인 후보를 범주화하고, 비용이 낮은 확인부터 차례대로 지우는 방식이 필요하다.

내가 한 일은 모호한 SDK 실패를 재현 가능한 점검 순서로 바꾼 것이다. 이 방식이 있어야 다음 장애에서도 같은 로그를 보고 처음부터 다시 헤매지 않을 수 있다.
