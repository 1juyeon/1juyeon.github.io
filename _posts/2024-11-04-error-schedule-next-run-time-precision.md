## 문제

비밀번호 변경 스케줄과 DB 백업 스케줄에서 특정 주기 설정 시 작업이 1분 일찍 실행되는 문제가 있었다.

임의 기간, 분기, 반기 같은 주기를 설정할 때 다음 실행 시간이 저장되는데, 이 값의 초/밀리초 정밀도가 기존 시간 비교 로직과 맞지 않았다.

## 내가 확인한 것

스케줄 실행 여부를 판단하는 흐름을 따라가며 확인했다.

- `next_run_time`이 어떤 형식으로 저장되는지
- 주기별 계산 로직이 초와 밀리초를 어떻게 다루는지
- 현재 시간과 다음 실행 시간을 비교할 때 보정값이 들어가는지
- 기존 시간에 1분을 더해 비교하는 로직과 충돌하지 않는지
- 비밀번호 변경 스케줄과 백업 스케줄이 같은 기준을 쓰는지

문제는 저장되는 다음 실행 시간의 정밀도와 비교 로직의 정밀도가 어긋나는 데 있었다.

## 수정 방향

수정은 다음 실행 시간을 만들 때 비교 기준과 같은 정밀도로 맞추는 방향으로 했다.

```text
주기 선택 -> 다음 실행 시간 계산 -> 초/밀리초 기준 정리 -> 비교 로직 적용
```

스케줄은 1분 차이도 운영에서는 크게 보일 수 있다. 특히 백업이나 비밀번호 변경은 정해진 시간에 실행된다는 신뢰가 중요하다.

## 검증한 케이스

아래 주기들을 나눠 확인했다.

- 매일 실행
- 임의 기간 실행
- 분기 실행
- 반기 실행
- 비밀번호 변경 스케줄
- DB 백업 스케줄

각 케이스에서 저장된 다음 실행 시간과 실제 실행 판단 시간이 일치하는지 봤다.

## 정리

스케줄 오류는 로직이 크게 틀려서 생기기보다 시간 정밀도 차이에서 생길 수 있다.

날짜만 맞는지 보지 말고, 시/분/초/밀리초가 비교 로직과 같은 기준으로 저장되는지 확인해야 한다.
