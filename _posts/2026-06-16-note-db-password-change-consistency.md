## 이번에 한 작업

관리자가 웹 화면에서 민감한 운영 설정값을 바꿀 수 있는 기능을 구현했다.

처음 요구사항은 단순해 보였다. "DB 접속 정보를 바꾸고, 설정 파일에도 암호화해서 저장한다." 하지만 코드를 따라가 보니 이 값은 한 군데에만 있지 않았다.

내가 실제로 같이 맞춰야 했던 것은 다음과 같았다.

- DB 계정의 실제 비밀번호
- 애플리케이션이 읽는 DB 접속 설정 파일
- DB 백업 스크립트에서 사용하는 인증 정보
- 변경 이력 테이블
- 설정 파일 백업본
- 변경 후 서비스 재기동 흐름

하나라도 빠지면 화면에서는 변경 성공처럼 보여도 다음 기동이나 백업 시점에 장애가 날 수 있었다.

구현 단위로는 아래를 같이 수정했다.

- 관리자 설정 화면에 민감 설정 변경 영역 추가
- 변경 요청을 받는 컨트롤러와 서비스 흐름 추가
- 설정 변경 이력 테이블, 모델, 매퍼 추가
- 설정 파일을 백업하고 `ENC(...)` 값으로 다시 쓰는 처리 추가
- 백업 스크립트의 인증 정보 동기화 처리 추가
- 변경 후 재기동 안내와 실패 위치별 결과 메시지 정리

## 구현 흐름

내가 잡은 처리 순서는 이랬다.

1. 관리자 권한과 관리자 비밀번호를 다시 확인한다.
2. 새 비밀번호를 랜덤 생성하거나 지정값으로 받는다.
3. 허용 문자와 길이를 검증한다.
4. DB 계정 비밀번호를 변경한다.
5. 새 비밀번호로 새 연결을 열어 `SELECT 1` 검증을 한다.
6. 설정 파일을 백업한 뒤 암호화된 값으로 교체한다.
7. 백업 스크립트의 비밀번호도 동기화한다.
8. 변경 이력을 저장한다.
9. 적용을 위해 서비스 재기동 흐름으로 넘긴다.

중요한 건 5번이었다. DB 계정 변경이 성공했다고 끝내지 않고, 새 비밀번호로 실제 연결을 다시 열어 확인했다.

## 실패 처리

이 기능에서 가장 조심한 부분은 중간 실패였다.

예를 들어 DB 비밀번호는 바뀌었는데 설정 파일 기록에 실패하면, 다음 기동 때 애플리케이션은 DB에 붙지 못한다. 그래서 실패 위치별로 대응을 나눴다.

| 실패 위치 | 처리 |
| --- | --- |
| 입력 검증 실패 | 변경 시작 전이므로 이력 없이 중지 |
| DB 변경 실패 | 기존 상태 유지 |
| 새 연결 검증 실패 | DB 비밀번호 원복 |
| 설정 파일 기록 실패 | DB 비밀번호 원복 |
| 백업 스크립트 동기화 실패 | 비밀번호 변경은 성공으로 두되 경고 표시 |
| 원복 실패 | 수동 조치 필요 메시지를 이력에 남김 |

모든 실패를 같은 오류로 묶지 않고, 어느 단계에서 실패했는지 이력 결과에 남기도록 했다.

## 화면에서 신경 쓴 부분

화면도 단순 입력 폼으로 끝내지 않았다.

- 현재 비밀번호와 이력 비밀번호는 기본 마스킹
- 보기 버튼을 눌렀을 때만 관리자 재인증 후 조회
- 변경 이력은 최신 5개만 표시
- 페이징은 제거
- 비밀번호 표시 영역과 눈/복사 버튼 위치가 비밀번호 길이에 따라 흔들리지 않게 조정
- 변경 버튼 클릭 시 필수 파일이 없으면 안내 후 중지

이 부분은 기능보다 사용 실수 방지가 목적이었다. DB 비밀번호 변경은 한 번 잘못 누르면 영향이 크기 때문에, 화면에서 최대한 의도를 명확하게 보여줘야 했다.

## 백업 정책 변경

처음에는 설정 파일 백업을 누적하는 방향도 검토했지만, 운영상 최신본 하나를 복원하기 쉽게 두는 쪽으로 정리했다.

- 애플리케이션 내부에는 최신 백업 1개만 유지
- 외부 백업 위치에는 최신 설정 파일을 단일 파일로 덮어쓰기
- 재패치로 애플리케이션 폴더가 바뀌어도 복원할 수 있게 함

백업이 실패해도 비밀번호 변경 자체는 성공할 수 있도록 비치명 경고로 처리했다. 대신 설정 파일 기록 자체는 실패 시 원복 대상으로 봤다.

## 정리

이번 작업은 `ALTER USER`를 실행하는 기능이 아니라, DB 계정과 애플리케이션 설정과 백업 스크립트를 같은 상태로 맞추는 기능이었다.

운영 설정 변경처럼 파일 작업과 DB 작업이 섞이는 기능은 완전한 트랜잭션으로 묶기 어렵다. 그래서 단계별 검증, 원복, 이력 메시지, 백업 정책을 같이 설계해야 한다.
