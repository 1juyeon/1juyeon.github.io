# 네트워크 장비 비밀번호 변경을 JSON 템플릿으로 확장한 기록

## 이번에 한 작업

네트워크 장비 비밀번호 변경 기능을 제조사별 코드만으로 처리하지 않고, JSON 템플릿으로도 확장할 수 있게 보완했다.

처음에는 특정 제조사마다 클래스를 만들어 처리하는 방식이 자연스러웠다. 하지만 현장에서는 콘솔 명령 형식이 조금씩 다른 장비가 계속 나오고, 모든 장비를 코드 배포로만 대응하기 어렵다.

그래서 템플릿 기반 흐름을 추가했다.

내가 실제로 손댄 부분은 다음과 같았다.

- 장비 비밀번호 변경 명령을 JSON 템플릿으로 정의할 수 있게 추가
- 현장에서 콘솔 형식을 확인한 뒤 템플릿을 등록해 테스트할 수 있게 수정
- 제조사별 전용 클래스와 공통 템플릿 방식을 구분하는 값 추가
- 장비 등록 시 비밀번호 길이 제한과 지정 규칙 길이 제한을 다시 조정
- 템플릿 작성 방법 문서화
- JSON 오류와 예외 케이스 보완

## 템플릿으로 바꾼 이유

네트워크 장비 자동화는 제조사명만으로 분기하기 어렵다.

같은 제조사라도 모델, 펌웨어, CLI 모드에 따라 명령 흐름이 다를 수 있다. 그래서 코드에 모든 케이스를 박아 넣으면 배포 주기가 현장 대응 속도를 따라가지 못한다.

템플릿 방식은 이런 상황에서 유리했다.

```text
접속 -> 현재 모드 확인 -> 명령 입력 -> 응답 패턴 확인 -> 다음 명령 입력 -> 저장
```

이 흐름을 JSON으로 표현하면, 간단한 장비는 코드 수정 없이도 추가할 수 있다.

## 구현하면서 나눈 기준

모든 장비를 템플릿으로만 처리하지는 않았다.

복잡한 예외 처리나 상태 판정이 필요한 장비는 전용 클래스로 두고, 일반적인 콘솔 흐름으로 처리 가능한 장비는 템플릿으로 다루는 기준을 잡았다.

| 방식 | 적합한 경우 |
| --- | --- |
| 전용 클래스 | 예외가 많고 상태 판정이 복잡한 장비 |
| JSON 템플릿 | 명령/응답 흐름이 비교적 단순한 장비 |

## 검증한 것

템플릿 방식은 작성 오류가 바로 런타임 오류로 이어질 수 있다.

그래서 다음 항목을 확인했다.

- JSON 문법 오류가 명확히 드러나는지
- 필수 필드가 없을 때 안내되는지
- 응답 패턴이 맞지 않을 때 다음 단계로 넘어가지 않는지
- 비밀번호 변경 실패와 접속 실패가 구분되는지
- 템플릿 추가 후 기존 제조사별 로직이 깨지지 않는지

## 정리

네트워크 장비 자동화는 코드로 모든 장비를 직접 구현하는 방식만으로는 확장성이 부족하다.

전용 클래스와 JSON 템플릿을 같이 두면, 복잡한 장비는 코드로 안정성을 확보하고 단순한 장비는 현장 대응 속도를 높일 수 있다. 중요한 건 템플릿도 운영 코드의 일부로 보고 검증과 문서화를 같이 하는 것이다.
