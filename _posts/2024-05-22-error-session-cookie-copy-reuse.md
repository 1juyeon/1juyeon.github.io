## 문제

같은 IP에서 서로 다른 브라우저로 각각 다른 계정에 로그인한 뒤, 한쪽 세션 값을 다른 브라우저에 복사하면 세션을 복사한 계정 쪽으로 접근 가능한 문제가 있었다.

처음에는 같은 IP 조건을 추가하면 막을 수 있을 것처럼 보였다. 하지만 같은 PC나 같은 네트워크에서 테스트하면 IP는 동일하기 때문에 IP만으로는 충분하지 않았다.

## 내가 확인한 것

확인한 지점은 인증 인터셉터와 세션 검증 흐름이었다.

- 요청마다 세션의 사용자 정보가 어떻게 확인되는지
- DB나 캐시에 저장된 로그인 상태와 현재 세션이 맞는지
- IP 조건만으로 세션 재사용을 막고 있는지
- 브라우저가 달라도 세션 쿠키가 같으면 같은 사용자로 처리되는지
- 로그아웃이나 재로그인 시 기존 세션 정보가 정리되는지

## 수정 방향

수정 방향은 "IP가 같으면 허용"이 아니라 "현재 요청의 세션이 서버가 기억하는 로그인 상태와 일치해야 허용"으로 바꿨다.

즉, 세션 ID만 복사했다고 해서 인증된 요청으로 보지 않고, 서버 측에 저장된 사용자별 세션 정보와 함께 비교하도록 보완했다.

```text
요청 세션 확인 -> 사용자 로그인 상태 조회 -> 세션/사용자 매칭 검증 -> 통과 시 처리
```

## 같이 보완한 것

이 작업을 하면서 전체 기능에 보안 검증용 로그도 추가했다.

세션 관련 문제는 화면만 보면 정상 로그인처럼 보일 수 있어서, 어떤 사용자와 어떤 세션이 매칭됐는지 추적할 수 있어야 했다. 다만 로그에는 세션 값 자체나 민감 정보가 그대로 남지 않도록 주의해야 했다.

## 정리

세션 재사용 방지는 IP 비교만으로는 부족하다.

특히 같은 IP에서 여러 브라우저를 쓰는 상황은 충분히 현실적이다. 인증 검증은 클라이언트가 들고 온 쿠키뿐 아니라 서버가 관리하는 로그인 상태와 함께 확인해야 한다.
