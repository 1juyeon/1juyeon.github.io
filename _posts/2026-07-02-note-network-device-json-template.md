네트워크 장비의 비밀번호를 바꾸는 기능은 처음에 제조사별 Java 클래스로 구현되어 있었다. 새 장비가 들어올 때마다 접속, 명령 전송, 프롬프트 대기, 오류 문자열 분류를 코드에 추가해야 했고 작은 명령 차이에도 다시 빌드해야 했다.

현장에서 확인한 CLI 흐름을 JSON 파일로 옮기면 단순한 장비는 코드 배포 없이 템플릿만 추가할 수 있다고 판단했다. 그래서 전용 구현은 유지하면서 공통 SSH 템플릿 엔진을 새로 만들었다.

## 코드로 고정돼 있던 흐름

SSH 기반 장비 제어는 단일 명령을 보내는 작업이 아니었다.

```text
SSH 로그인
-> 현재 프롬프트 확인
-> 관리자 모드 진입
-> 설정 모드 진입
-> 비밀번호 변경 명령
-> 설정 저장
-> 새 비밀번호로 재접속 확인
```

장비마다 `>`, `#`, `(config)#` 같은 프롬프트와 저장 명령이 달랐다. 이 차이를 각 클래스의 `if/else`로 늘리는 대신 명령과 기대 응답을 데이터로 분리했다.

## 템플릿에 표현한 값

템플릿은 연결 설정과 실행 단계를 나눴다. 아래는 구조를 설명하기 위해 장비 이름과 실제 명령을 바꾼 예시다.

```json
{
  "info": {
    "ssh_port": 22,
    "ssh_timeout_ms": 10000,
    "expect_timeout_sec": 15
  },
  "change_pw": [
    {
      "name": "wait_prompt",
      "expect_any": [">", "#"]
    },
    {
      "name": "change_password",
      "send": "account ${user_id} secret ${new_password}",
      "sensitive": true,
      "expect": "#",
      "unexpected_result": 9012
    }
  ]
}
```

`send`는 보낼 명령, `expect`와 `expect_any`는 다음 단계로 넘어가기 전에 기다릴 문자열이다. `${user_id}`와 `${new_password}` 같은 토큰은 실행 직전에 실제 값으로 치환한다.

`sensitive: true`인 단계는 명령 전체를 로그에 남기지 않았다. 비밀번호가 포함되지 않은 출력도 혹시 기존·신규 비밀번호 문자열을 포함하면 `******`로 바꿔 기록했다.

## 실행 엔진을 만든 방식

Java에서는 JSch로 SSH 세션을 열고 ExpectIt으로 프롬프트를 기다렸다. 각 JSON 단계를 순서대로 읽으며 조건, 명령, 기대 응답을 처리했다.

```java
for (Step step : steps) {
    if (!shouldRun(step, lastOutput)) {
        continue;
    }

    sendResolvedCommand(step);
    String output = waitForExpectedPrompt(step);

    int result = classify(step, output);
    if (result != SUCCESS) {
        return result;
    }
}
```

선택 단계는 실패해도 다음으로 넘어가고, 필수 단계는 예상한 프롬프트가 오지 않으면 시간 초과 또는 템플릿에 지정된 오류 코드로 끝내도록 했다.

SSH 세션이 장비의 동시 접속 제한 때문에 끊긴 것으로 분류되면 2초 뒤 한 번 다시 연결했다. 반면 인증 실패나 네트워크 시간 초과는 같은 재시도로 해결될 가능성이 낮아 별도 코드로 반환했다.

## 오류 문자열의 우선순위

장비 CLI는 HTTP처럼 구조화된 오류 코드를 주지 않고 문자열을 출력하는 경우가 많다. 그래서 템플릿의 `error_map`에 실측한 문자열과 애플리케이션 오류 코드를 연결했다.

```json
"error_map": [
  {
    "contains_any": ["recently used", "previous password"],
    "result_code": 1016
  },
  {
    "contains_any": ["too short", "minimum", "complex"],
    "result_code": 9006
  }
]
```

구체적인 오류 규칙을 먼저 검사하고 일반 규칙을 나중에 적용했다. 실제로 어떤 장비는 `%`로 시작하는 정상 저장 메시지를 보냈는데, `%`가 있으면 정책 오류라고 보는 일반 규칙 때문에 성공을 실패로 오판했다. 이후 `saving`, `copy complete`, `[OK]` 같은 확인된 정상 표시는 예외로 분리했다.

## 전용 클래스와 공통 엔진의 경계

모든 장비를 JSON으로 바꾸지는 않았다.

| 구현 방식 | 선택 기준 |
| --- | --- |
| 전용 Java 클래스 | 상태 분기가 복잡하고 별도 API나 예외 처리가 많은 경우 |
| 공통 JSON 템플릿 | SSH 명령과 프롬프트의 순서로 표현할 수 있는 경우 |

팩토리에서는 알려진 제조사의 전용 클래스를 먼저 사용하고, 매칭되지 않으면서 템플릿 이름이 있는 장비를 공통 엔진으로 보냈다. 이후 같은 제조사 안에서도 새 모델만 공통 엔진으로 보내야 해서 DB에 실행 방식 선택 값을 추가했다.

## 검증하며 확인한 한계

- JSON 문법이 맞아도 명령과 프롬프트가 실제 펌웨어와 다르면 실패한다.
- `StrictHostKeyChecking=no` 같은 호환 설정은 편하지만 호스트 키 검증을 포기하므로 폐쇄망 조건과 별도 위험 판단이 필요하다.
- 비밀번호 변경 성공은 명령 출력만이 아니라 새 비밀번호 재로그인으로 확인해야 한다.
- 템플릿 파일과 DB 매핑이 함께 배포되지 않으면 실행 경로 자체가 만들어지지 않는다.
- 실제 장비 없이 파서와 분기만 통과한 결과를 현장 검증 완료로 보지 않았다.

이 작업에서 가장 시간이 들었던 부분은 JSON 파서가 아니라 여러 장비의 모호한 문자열을 성공, 인증 실패, 정책 실패, 세션 혼잡으로 나누는 기준이었다. 템플릿을 설정 파일이 아니라 실행 로직의 일부로 보고 로그 마스킹, 오류 우선순위, 로드 실패 처리까지 함께 구현했다.
