# ChatGPT 데이터 내보내기를 세션 아카이브에 연결한 기록

## 이번에 한 작업

로컬 AI 세션 아카이브가 이미 여러 코딩 도구의 로그를 Markdown으로 모으고 있었지만, ChatGPT 대화는 앱에서 바로 읽는 방식으로 다루기 어려웠다. 이번에는 사용자가 내려받은 데이터 내보내기 ZIP을 입력으로 받아, 기존 아카이브와 같은 세션 모델과 폴더 구조로 변환하는 source adapter를 추가했다.

목표는 ZIP 안의 내용을 그대로 복사하는 것이 아니었다. 대화 목록, 메시지 트리, 이미지 첨부를 읽고, 기존 저장 형식에 맞춘 뒤 기본적인 민감정보 치환 규칙을 거쳐 다시 찾아볼 수 있는 기록으로 남기는 쪽에 맞췄다.

## ZIP 하나에 들어 있는 대화를 세션 단위로 풀기

내보내기 파일에는 대화가 하나의 JSON 파일만으로 끝나지 않는 경우가 있다. 그래서 exporter가 ZIP을 열어 대화 JSON 조각을 모두 찾고, 각 대화를 독립된 세션으로 정규화하도록 만들었다.

세션에는 대화 제목과 생성·수정 시각, 대화 식별자, 사용자와 어시스턴트 메시지를 담았다. 메시지 순서는 단순히 JSON 객체를 순회하지 않고 현재 대화 노드에서 부모 노드를 거슬러 올라가 복원했다. 이 과정을 넣지 않으면 분기된 대화에서 문답 순서가 뒤섞일 수 있었다.

같은 대화가 여러 입력에서 발견될 가능성도 있어 source, 대화 식별자, 원본 경로를 기준으로 최신 항목 하나만 남기도록 정리했다. 덕분에 여러 ZIP을 한 번에 처리해도 파일이 불필요하게 중복되지 않는다.

## 이미지는 본문 위치를 유지하고 별도 파일로 보관하기

텍스트만 저장하면 당시 어떤 입력을 보고 답했는지 잃어버릴 때가 있다. 특히 이미지가 포함된 대화는 문맥이 크게 줄어든다. 그래서 이미지 포인터를 만났을 때는 본문에 임시 토큰을 남기고, ZIP 안의 실제 바이너리를 꺼내 별도 assets 디렉터리에 저장한 뒤 그 위치를 Markdown 이미지 링크로 치환했다.

이미지 이름이나 확장자가 신뢰할 수 없는 경우를 고려해, 알려진 파일명뿐 아니라 파일 헤더도 확인해 미디어 형식을 정했다. 원본이 없거나 지원하지 않는 첨부 형식이면 내보내기 전체를 실패시키지 않고, 해당 위치에 누락되었음을 표시하는 쪽을 택했다.

## 실행 경로를 분리한 이유

기존 명령과 섞지 않고 `export-chatgpt` 명령을 추가했다. ZIP 경로는 명시적으로 받아야 하며, 여러 파일을 넘길 수도 있다. 윈도우에서 자주 쓰는 흐름을 위해 같은 인자를 전달하는 PowerShell·배치 래퍼도 두었다.

이 분리는 의도적이다. 다른 도구의 로컬 로그처럼 자동 탐색하지 않으므로, 내보내기 파일을 준비한 사람이 입력 범위를 한 번 더 확인할 수 있다. 기본값에서도 민감한 키워드와 토큰 형태는 치환하며, 공개 저장소가 아니라 개인 아카이브에서 먼저 검토하는 흐름은 그대로 유지했다.

## 검증한 내용

작은 테스트 ZIP을 만들어 대화 JSON, 이미지 자산 목록, PNG 파일을 함께 넣고 전체 변환을 실행했다. 다음을 확인했다.

- 사용자와 어시스턴트 메시지가 시간 순서로 저장되는지
- 이미지가 Markdown 본문 위치에 링크되고 assets 디렉터리에 실제 파일로 생성되는지
- PNG 헤더가 보존되는지
- 기존 Claude Code·Codex 변환 테스트가 함께 깨지지 않는지

전체 테스트는 8건 모두 통과했다. 특히 첨부를 단순 base64 텍스트로 본문에 남기지 않아, Markdown 파일이 과도하게 커지지 않는 점을 다시 확인했다.

## 정리

이번 변경으로 대화 기록의 입력 형식 하나가 더 늘었지만, 핵심은 새 포맷을 추가하는 것보다 기존의 보수적인 저장 원칙을 유지한 데 있다. 대화 순서, 중복, 첨부 누락처럼 실제 내보내기에서 흔한 예외를 먼저 흡수하고, 원본 범위는 로컬에서 검토한 뒤 아카이브에 남기도록 정리했다.
